662=head1 NAME
663
664POE::Wheel::FollowTail - follow the tail of an ever-growing file
665
666=head1 SYNOPSIS
667
668 #!perl
669
670 use POE qw(Wheel::FollowTail);
671
672 POE::Session->create(
673 inline_states => {
674 _start => sub {
675 $_[HEAP]{tailor} = POE::Wheel::FollowTail->new(
676 Filename => "/var/log/system.log",
677 InputEvent => "got_log_line",
678 ResetEvent => "got_log_rollover",
679 );
680 },
681 got_log_line => sub {
682 print "Log: $_[ARG0]\n";
683 },
684 got_log_rollover => sub {
685 print "Log rolled over.\n";
686 },
687 }
688 );
689
690 POE::Kernel->run();
691 exit;
692
693=head1 DESCRIPTION
694
695POE::Wheel::FollowTail objects watch for new data at the end of a file
696and generate new events when things happen to the file. Its C<Filter>
697parameter defines how to parse data from the file. Each new item is sent
698to the creator's session as an C<InputEvent> event. Log rotation will
699trigger a C<ResetEvent>.
700
701POE::Wheel::FollowTail only reads from a file, so it doesn't implement
702a put() method.
703
704=head1 PUBLIC METHODS
705
706=head2 new
707
708new() returns a new POE::Wheel::FollowTail object. As long as this
709object exists, it will generate events when the corresponding file's
710status changes.
711
712new() accepts a small set of named parameters:
713
714=head3 Driver
715
716The optional C<Driver> parameter specifies which driver to use when
717reading from the tailed file. If omitted, POE::Wheel::FollowTail will
718use POE::Driver::SysRW. This is almost always the right thing to do.
719
720=head3 Filter
721
722C<Filter> is an optional constructor parameter that specifies how to
723parse data from the followed file. By default, POE::Wheel::FollowTail
724will use POE::Filter::Line to parse files as plain, newline-separated
725text.
726
727 $_[HEAP]{tailor} = POE::Wheel::FollowTail->new(
728 Filename => "/var/log/snort/alert",
729 Filter => POE::Filter::Snort->new(),
730 InputEvent => "got_snort_alert",
731 );
732
733=head3 PollInterval
734
735POE::Wheel::FollowTail needs to periodically check for new data on the
736followed file. C<PollInterval> specifies the number of seconds to
737wait between checks. Applications that need to poll once per second
738may omit C<PollInterval>, as it defaults to 1.
739
740Longer poll intervals may be used to reduce the polling overhead for
741infrequently updated files.
742
743 $_[HEAP]{tailor} = POE::Wheel::FollowTail->new(
744 ...,
745 PollInterval => 10,
746 );
747
748=head3 Seek
749
750If specified, C<Seek> instructs POE::Wheel::FollowTail to seek to a
751specific spot in the tailed file before beginning to read from it. A
752positive C<Seek> value is interpreted as the number of octets to seek
753from the start of the file. Negative C<Seek> will, like negative
754array indices, seek backwards from the end of the file. Zero C<Seek>
755starts reading from the beginning of the file.
756
757Be careful when using C<Seek>, as it's quite easy to seek into the
758middle of a record. When in doubt, and when beginning at the end of
759the file, omit C<Seek> entirely. POE::Wheel::FollowTail will seek
7604 kilobytes back from the end of the file, then parse and discard all
761records unto EOF. As long as the file's records are smaller than 4
762kilobytes, this will guarantee that the first record returned will be
763complete.
764
765C<Seek> may also be used with the wheel's tell() method to restore the
766file position after a program restart. Save the tell() value prior to
767exiting, and load and C<Seek> back to it on subsequent start-up.
768

771=head3 SeekBack
772
773C<SeekBack> behaves like the inverse of C<Seek>. A positive value
774acts like a negative C<Seek>. A negative value acts like a positive
775C<Seek>. A zero C<SeekBack> instructs POE::Wheel::FollowTail to begin
776at the very end of the file.
777
778C<Seek> and C<SeekBack> are mutually exclusive.
779
780See L</Seek> for caveats, techniques, and an explanation of the magic
781that happens when neither C<Seek> nor C<SeekBack> is specified.
782

785=head3 Handle
786
787POE::Wheel::FollowTail may follow a previously opened file C<Handle>.
788Unfortunately it cannot follow log resets this way, as it won't be
789able to reopen the file once it has been reset. Applications that
790must follow resets should use C<Filename> instead.
791
792C<Handle> is still useful for files that will never be reset, or for
793devices that require setup outside of POE::Wheel::FollowTail's
794purview.
795
796C<Handle> and C<Filename> are mutually exclusive. One of them is
797required, however.
798

801=head3 Filename
802
803Specify the C<Filename> to watch. POE::Wheel::FollowTail will wait
804for the file to appear if it doesn't exist. The wheel will also
805reopen the file if it disappears, such as when it has been reset or
806rolled over. In the case of a reset, POE::Wheel::FollowTail will also

815
816The C<InputEvent> parameter is required, and it specifies the event to

885=head1 PUBLIC EVENTS
886
887POE::Wheel::FollowTail emits a small number of events.
888
889=head2 InputEvent
890
891C<InputEvent> sets the name of the event to emit when new data arrives
892into the tailed file. The event will be accompanied by two
893parameters:
894
895C<$_[ARG0]> contains the data that was read from the file, after being
896parsed by the current C<Filter>.
897
898C<$_[ARG1]> contains the wheel's ID, which may be used as a key into a
899data structure tracking multiple wheels. No assumption should be made
900about the nature or format of this ID, as it may change at any time.
901Therefore, track your wheels in a hash.
902
903See the L</SYNOPSIS> for an example.
904
905=head2 ResetEvent
906
907C<ResetEvent> names the event to be emitted whenever the wheel detects
908that the followed file has been reset. It's only available when
909watching files by name, as POE::Wheel::FollowTail must reopen the file
910after it has been reset.
911
912C<ResetEvent> comes with only one parameter, C<$_[ARG0]>, which
913contains the wheel's ID. See L</InputEvent> for some notes about what
914may be done with wheel IDs.
915
916See the L</SYNOPSIS> for an example.
917
918=head2 ErrorEvent
919
920C<ErrorEvent> names the event emitted when POE::Wheel::FollowTail
921encounters a problem. Every C<ErrorEvent> comes with four parameters
922that describe the error and its situation:
923
924C<$_[ARG0]> describes the operation that failed. This is usually
925"read", since POE::Wheel::FollowTail spends most of its time reading
926from a file.
927
928C<$_[ARG1]> and C<$_[ARG2]> contain the numeric and stringified values
929of C<$!>, respectively. They will never contain EAGAIN (or its local
930equivalent) since POE::Wheel::FollowTail handles that error itself.
931
932C<$_[ARG3]> contains the wheel's ID, which has been discussed in
933L</InputEvent>.
934
935This error handler logs a message to STDERR and then shuts down the
936wheel. It assumes that the session is watching multiple files.
937
938 sub handle_tail_error {
939 my ($operation, $errnum, $errstr, $wheel_id) = @_[ARG0..ARG3];
940 warn "Wheel $wheel_id: $operation error $errnum: $errstr\n";
941 delete $_[HEAP]{tailors}{$wheel_id};
942 }
