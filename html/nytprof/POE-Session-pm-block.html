760=head1 NAME
761
762POE::Session - a generic event-driven task
763
764=head1 SYNOPSIS
765
766 use POE; # auto-includes POE::Kernel and POE::Session
767
768 POE::Session->create(
769 inline_states => {
770 _start => sub { $_[KERNEL]->yield("next") },
771 next => sub {
772 print "tick...\n";
773 $_[KERNEL]->delay(next => 1);
774 },
775 },
776 );
777
778 POE::Kernel->run();
779 exit;
780
781POE::Session can also dispatch to object and class methods through
782L</object_states> and L</package_states> callbacks.
783
784=head1 DESCRIPTION
785
786POE::Session and its subclasses translate events from POE::Kernel's
787generic dispatcher into the particular calling conventions suitable for
788application code. In design pattern parlance, POE::Session classes
789are adapters between L<POE::Kernel> and application code.
790
791The L<sessions|POE::Kernel/Sessions> that POE::Kernel manages are more
792like generic task structures. Unfortunately these two disparate
793concepts have virtually identical names.
794
795=head2 A note on nomenclature
796
797This documentation will refer to event handlers as "states" in certain
798unavoidable situations. Sessions were originally meant to be
799event-driven state machines, but their purposes evolved over time.
800Some of the legacy vocabulary lives on in the API for backward
801compatibility, however.
802
803Confusingly, L<POE::NFA> is a class for implementing actual
804event-driven state machines. Its documentation uses "state" in the
805proper sense.
806
807=head1 USING POE::Session
808
809POE::Session has two main purposes. First, it maps event names to the
810code that will handle them. Second, it maps a consistent event
811dispatch interface to those handlers.
812
813Consider the L</SYNOPSIS> for example. A POE::Session instance is
814created with two C<inline_states>, each mapping an event name
815("_start" and "next") to an inline subroutine. POE::Session ensures
816that C<$_[KERNEL]> and so on are meaningful within an event handler.
817
818Event handlers may also be object or class methods, using
819L</object_states> and L</package_states> respectively. The create()
820syntax is different than for C<inline_states>, but the calling
821convention is nearly identical.
822
823Notice that the created POE::Session object has not been saved to a
824variable. The new POE::Session object gives itself to POE::Kernel,
825which then manages it and all the resources it uses.
826
827It's possible to keep references to new POE::Session objects, but it's
828not usually necessary. If an application is not careful about
829cleaning up these references you will create circular references,
830which will leak memory when POE::Kernel would normally destroy the
831POE::Session object. It is recommended that you keep the session's
832C</ID> instead.
833
834=head2 POE::Session's Calling Convention
835
836The biggest syntactical hurdle most people have with POE is
837POE::Session's unconventional calling convention. For example:
838
839 sub handle_event {
840 my ($kernel, $heap, $parameter) = @_[KERNEL, HEAP, ARG0];
841 ...;
842 }
843
844Or the use of C</$_[KERNEL]>, C</$_[HEAP]> and C</$_[ARG0]> inline,
845as is done in most examples.
846
847What's going on here is rather basic. Perl passes parameters into
848subroutines or methods using the @_ array. C<KERNEL>, C<HEAP>,
849C<ARG0> and others are constants exported by POE::Session (which is
850included for free when a program uses POE).
851
852So C</$_[KERNEL]> is an event handler's KERNELth parameter.
853C<@_[HEAP, ARG0]> is a slice of @_ containing the HEAPth and ARG0th
854parameters.
855
856While this looks odd, it's perfectly plain and legal Perl syntax. POE
857uses it for a few reasons:
858
859=over 4
860
861=item 1
862
863In the common case, passing parameters in C<@_> is faster than passing
864hash or array references and then dereferencing them in the handler.
865
866=item 2
867
868Typos in hash-based parameter lists are either subtle run-time
869errors or requires constant run-time checking. Constants are either
870known at compile time, or are clear compile-time errors.
871
872=item 3
873
874Referencing C<@_> offsets by constants allows parameters to move
875in the future without breaking application code.
876
877=item 4
878
879Most event handlers don't need all of C<@_>. Slices allow handlers to
880use only the parameters they're interested in.
881
882=back
883
884=head2 POE::Session Parameters
885
886Event handlers receive most of their run-time context in up to nine
887callback parameters. POE::Kernel provides many of them.
888
889=head3 $_[OBJECT]
890
891C<$_[OBJECT]> is $self for event handlers that are an object method. It is
892the class (package) name for class-based event handlers. It is undef
893for plain coderef callbacks, which have no special C<$self>-ish value.
894
895C<OBJECT> is always zero, since C<$_[0]> is always C<$self> or C<$class>
896in object and class methods. Coderef handlers are called with
897an C<undef> placeholder in C<$_[0]> so that the other offsets remain valid.
898
899It's often useful for method-based event handlers to call other
900methods in the same object. C<$_[OBJECT]> helps this happen.
901
902 sub ui_update_everything {
903 my $self = $_[OBJECT];
904 $self->update_menu();
905 $self->update_main_window();
906 $self->update_status_line();
907 }
908
909You may also use method inheritance. Here we invoke
910$self->a_method(@_). Since Perl's C<<->>> operator unshifts $self
911onto the beginning of @_, we must first shift a copy off to maintain
912POE's parameter offsets:
913
914 sub a_method {
915 my $self = shift;
916 $self->SUPER::a_method( @_ );
917 # ... more work ...
918 }

920=head3 $_[SESSION]
921
922C<$_[SESSION]> is a reference to the current session object. This lets event
923handlers access their session's methods. Programs may also compare
924C<$_[SESSION]> to C</$_[SENDER]> to verify that intra-session events did not
925come from other sessions.
926
927C<$_[SESSION]> may also be used as the destination for intra-session
928C<post()|POE::Kernel/post> and C<call()|POE::Kernel/call>. C<yield()|POE::Kernel/yield> is marginally more convenient and
929efficient than C<post($_[SESSION], ...)> however.
930
931It is bad form to access another session directly. The recommended
932approach is to manipulate a session through an event handler.
933
934 sub enable_trace {
935 my $previous_trace = $_[SESSION]->option( trace => 1 );
936 my $id = $_[SESSION]->ID;
937 if ($previous_trace) {
938 print "Session $id: dispatch trace is still on.\n";
939 }
940 else {
941 print "Session $id: dispatch trace has been enabled.\n";
942 }
943 }

945=head3 $_[KERNEL]
946
947The KERNELth parameter is always a reference to the application's
948singleton L<POE::Kernel> instance. It is most often used to call
949POE::Kernel methods from event handlers.
950
951 # Set a 10-second timer.
952 $_[KERNEL]->delay( time_is_up => 10 );
953
954=head3 $_[HEAP]
955
956Every POE::Session object contains its own variable namespace known as
957the session's C<HEAP>. It is modeled and named after process memory
958heaps (not priority heaps). Heaps are by default anonymous hash
959references, but they may be initialized in L<create()|/create> to be almost
960anything. POE::Session itself never uses C<$_[HEAP]>, although some POE
961components do.
962
963Heaps do not overlap between sessions, although create()'s "heap"
964parameter can be used to make this happen.
965
966These two handlers time the lifespan of a session:
967
968 sub _start_handler {
969 $_[HEAP]{ts_start} = time();
970 }
971
972 sub _stop_handler {
973 my $time_elapsed = time() - $_[HEAP]{ts_start};
974 print "Session ", $_[SESSION]->ID, " elapsed seconds: $elapsed\n";
975 }

977=head3 $_[STATE]
978
979The STATEth handler parameter contains the name of the event being
980dispatched in the current callback. This can be important since the
981event and handler names may significantly differ. Also, a single
982handler may be assigned to more than one event.
983
984 POE::Session->create(
985 inline_states => {
986 one => \&some_handler,
987 two => \&some_handler,
988 six => \&some_handler,
989 ten => \&some_handler,
990 _start => sub {
991 $_[KERNEL]->yield($_) for qw(one two six ten);
992 }
993 }
994 );
995
996 sub some_handler {
997 print(
998 "Session ", $_[SESSION]->ID,
999 ": some_handler() handled event $_[STATE]\n"

1001 }

1003It should be noted however that having event names and handlers names match
1004will make your code easier to navigate.

1006=head3 $_[SENDER]
1007
1008Events must come from somewhere. C<$_[SENDER]> contains the currently
1009dispatched event's source.
1010
1011C<$_[SENDER]> is commonly used as a return address for responses. It may
1012also be compared against C<$_[KERNEL]> to verify that timers and other
1013POE::Kernel-generated events were not spoofed.

1015This C<echo_handler()> reponds to the sender with an "echo" event that
1016contains all the parameters it received. It avoids a feedback loop by
1017ensuring the sender session and event (STATE) are not identical to the
1018current ones.

1060=head2 Using POE::Session With Objects
1061
1062One session may handle events across many objects. Or looking at it
1063the other way, multiple objects can be combined into one session. And
1064what the heck---go ahead and mix in some inline code as well.
1065
1066 POE::Session->create(
1067 object_states => [
1068 $object_1 => { event_1a => "method_1a" },
1069 $object_2 => { event_2a => "method_2a" },
1070 ],
1071 inline_states => {
1072 event_3 => \&piece_of_code,
1073 },
1074 );
1075
1076However only one handler may be assigned to a given event name.
1077Duplicates will overwrite earlier ones.
1078
1079event_1a is handled by calling C<< $object_1->method_1a(...) >>. C<$_[OBJECT]>
1080is C<$object_1> in this case. C<$_[HEAP]> belongs to the session, which
1081means anything stored there will be available to any other event
1082handler regardless of the object.
1083
1084event_2a is handled by calling C<< $object_2->method_2a(...) >>. In this
1085case C</$_[OBJECT]> is $object_2. C<$_[HEAP]> is the same anonymous hashref
1086that was passed to the event_1a handler, though. The methods are resolved
1087when the event is handled (late-binding).
1088
1089event_3 is handled by calling C<piece_of_code(...)>. C<$_[OBJECT]> is C<undef>
1090here because there's no object. And once again, C<$_[HEAP]> is the same
1091shared hashref that the handlers for event_1a and event_2a saw.
1092
1093Interestingly, there's no technical reason that a
1094single object can't handle events from more than one session:
1095
1096 for (1..2) {
1097 POE::Session->create(
1098 object_states => [
1099 $object_4 => { event_4 => "method_4" },
1100 ]
1101 );
1102 }
1103
1104Now C<< $object_4->method_4(...) >> may be called to handle events from one of
1105two sessions. In both cases, C<$_[OBJECT]> will be C<$object_4>, but
1106C<$_[HEAP]> will hold data for a particular session.
1107
1108The same goes for inline states. One subroutine may handle events
1109from many sessions. C<$_[SESSION]> and C<$_[HEAP]> can be used within the
1110handler to easily access the context of the session in which the event
1111is being handled.

1274C<package_states> associates one or more classes to a session and maps
1275event names to the class methods that will handle them. Its function
1276is analogous to C<object_states>, but package names are specified
1277rather than objects.
1278
1279In fact, the following documentation is a copy of the C<object_states>
1280description with some word substitutions.
1281
1282The value for C<package_states> is an B<ARRAYREF> to be consistent
1283with C<object_states>, even though class names (also known as package names) are
1284already strings, so it's not necessary to avoid stringifying them.
1285
1286Here _start is handled by C<< $class_name->_session_start() >> and _stop
1287triggers C<< $class_name->_session_stop() >>:
1288
1289 POE::Session->create(
1290 package_states => [
1291 $class_name => {
1292 _start => '_session_start',
1293 _stop => '_session_stop',
1294 }
1295 ]
1296 );
1297
1298POE::Session also supports a short form where the event and method
1299names are identical. Here _start invokes C<< $class_name->_start() >>, and
1300_stop triggers C<< $class_name->_stop() >>:
1301
1302 POE::Session->create(
1303 package_states => [
1304 $class_name => [ '_start', '_stop' ],
1305 ]
1306 );

1308=head2 ID
1309
1310C<ID()> returns the session instance's unique identifier. This is an
1311integer that starts at 1 and counts up forever, or until the number
1312wraps around.
1313
1314It's theoretically possible that a session ID will not be unique, but
1315this requires at least 4.29 billion sessions to be created within a
1316program's lifespan. POE guarantees that no two sessions will have the
1317same ID at the same time, however; your computer doesn't have enough memory
1318to store 4.29 billion session objects.
1319
1320A session's ID is unique within a running process, but multiple
1321processes are likely to have the same session IDs. If a global ID is
1322required, it will need to include both C<< $_[KERNEL]->ID >> and
1323C<< $_[SESSION]->ID >>.

1325=head2 option OPTION_NAME [, OPTION_VALUE [, OPTION_NAME, OPTION_VALUE]... ]
1326
1327C<option()> sets and/or retrieves the values of various session options.
1328The options in question are implemented by POE::Session and do not
1329have any special meaning anywhere else.
1330
1331It may be called with a single OPTION_NAME to retrieve the value of
1332that option.
1333
1334 my $trace_value = $_[SESSION]->option('trace');
1335
1336C<option()> sets an option's value when called with a single OPTION_NAME,
1337OPTION_VALUE pair. In this case, C<option()> returns the option's
1338previous value.
1339
1340 my $previous_trace = $_[SESSION]->option(trace => 1);
1341
1342C<option()> may also be used to set the values of multiple options at
1343once. In this case, C<option()> returns all the specified options'
1344previous values in an anonymous hashref:
1345
1346 my $previous_values = $_[SESSION]->option(
1347 trace => 1,
1348 debug => 1,
1349 );
1350
1351 print "Previous option values:\n";
1352 while (my ($option, $old_value) = each %$previous_values) {
1353 print " $option = $old_value\n";
1354 }

1357
1358=head3 The "debug" option.
1359
1360The "debug" option is intended to enable additional warnings when
1361strange things are afoot within POE::Session. At this time, there is
1362only one additional warning:
1363
1364=over 4
1365
1366=item *
1367
1368Redefining an event handler does not usually cause a warning, but it
1369will when the "debug" option is set.
1370
1371=back
1372
1373=head3 The "default" option.
1374
1375Enabling the "default" option causes unknown events to become
1376warnings, if there is no _default handler to catch them.
1377
1378The class-level C<POE::Session::ASSERT_STATES> flag is implemented by
1379enabling the "default" option on all new sessions.
1380
1381=head3 The "trace" option.
1382
1383Turn on the "trace" option to dump a log of all the events dispatched
1384to a particular session. This is a session-specific trace option that
1385allows individual sessions to be debugged.
1386
1387Session-level tracing also indicates when events are redirected to
1388_default. This can be used to discover event naming errors.
1389
1390=head3 User-defined options.
1391
1392C<option()> does not verify whether OPTION_NAMEs are known, so C<option()>
1393may be used to store and retrieve user-defined information.
1394
1395Choose option names with caution. There is no established convention
1396to avoid namespace collisions between user-defined options and future
1397internal options.

1399=head2 postback EVENT_NAME, EVENT_PARAMETERS
1400
1401C<postback()> manufactures callbacks that post POE events. It returns an
1402anonymous code reference that will post EVENT_NAME to the target
1403session, with optional EVENT_PARAMETERS in an array reference in ARG0.
1404Parameters passed to the callback will be sent in an array reference
1405in ARG1.
1406
1407In other words, ARG0 allows the postback's creator to pass context
1408through the postback. ARG1 allows the caller to return information.
1409
1410This example creates a coderef that when called posts "ok_button" to
1411C<$some_session> with ARG0 containing C<[ 8, 6, 7 ]>.
1412
1413 my $postback = $some_session->postback( "ok_button", 8, 6, 7 );
1414
1415Here's an example event handler for "ok_button".
1416
1417 sub handle_ok_button {
1418 my ($creation_args, $called_args) = @_[ARG0, ARG1];
1419 print "Postback created with (@$creation_args).\n";
1420 print "Postback called with (@$called_args).\n";
1421 }
1422
1423Calling $postback->(5, 3, 0, 9) would perform the equivalent of...
1424
1425 $poe_kernel->post(
1426 $some_session, "ok_button",
1427 [ 8, 6, 7 ],
1428 [ 5, 3, 0, 9 ]
1429 );
1430
1431This would be displayed when "ok_button" was dispatched to
1432handle_ok_button():
1433
1434 Postback created with (8 6 7).
1435 Postback called with (5 3 0 9).
1436
1437Postbacks hold references to their target sessions. Therefore
1438sessions with outstanding postbacks will remain active. Under every
1439event loop except Tk, postbacks are blessed so that DESTROY may be
1440called when their users are done. This triggers a decrement on their
1441reference counts, allowing sessions to stop.
1442
1443Postbacks have one method, weaken(), which may be used to reduce their
1444reference counts upon demand. weaken() returns the postback, so you
1445can do:
1446
1447 my $postback = $session->postback("foo")->weaken();
1448
1449Postbacks were created as a thin adapter between callback libraries
1450and POE. The problem at hand was how to turn callbacks from the Tk
1451graphical toolkit's widgets into POE events without subclassing
1452several Tk classes. The solution was to provide Tk with plain old
1453callbacks that posted POE events.
1454
1455Since C<postback()> and C<callback()> are Session methods, they may be
1456called on C<$_[SESSION]> or C<$_[SENDER]>, depending on particular needs.
1457There are usually better ways to interact between sessions than
1458abusing postbacks, however.
1459
1460Here's a brief example of attaching a L<Gtk2> button to a POE event
1461handler:
1462
1463 my $btn = Gtk2::Button->new("Clear");
1464 $btn->signal_connect( "clicked", $_[SESSION]->postback("ev_clear") );
1465
1466Points to remember: The session will remain alive as long as $btn
1467exists and holds a copy of $_[SESSION]'s postback. Any parameters
1468passed by the Gtk2 button will be in ARG1.
1469
1470=head2 callback EVENT_NAME, EVENT_PARAMETERS
1471
1472callback() manufactures callbacks that use C<< $poe_kernel->call() >> to
1473deliver POE events rather than C<< $poe_kernel->post() >>. It is identical
1474to C<postback()> in every other respect.
1475
1476callback() was created to avoid race conditions that arise when
1477external libraries assume callbacks will execute synchronously.
1478L<File::Find> is an obvious (but not necessarily appropriate) example.
1479It provides a lot of information in local variables that stop being
1480valid after the callback. The information would be unavailable by the
1481time a post()ed event was dispatched.

1483=head2 get_heap
1484
1485C<get_heap()> returns a reference to a session's heap. This is the same
1486value as C<$_[HEAP]> for the target session. C<get_heap()> is intended to
1487be used with C<$poe_kernel> and POE::Kernel's C<get_active_session()> so
1488that libraries do not need these three common values explicitly passed
1489to them.
1490
1491That is, it prevents the need for:
1492
1493 sub some_helper_function {
1494 my ($kernel, $session, $heap, @specific_parameters) = @_;
1495 ...;
1496 }
1497
1498Rather, helper functions may use:
1499
1500 use POE::Kernel; # exports $poe_kernel
1501 sub some_helper_function {
1502 my (@specific_parameters) = @_;
1503 my $session = $poe_kernel->get_active_session();
1504 my $heap = $session->get_heap();
1505 }
1506
1507This isn't very convenient for people writing libraries, but it makes
1508the libraries much more convenient to use.
1509
1510Using C<get_heap()> to break another session's encapsulation is strongly
1511discouraged.

1513=head2 instantiate CREATE_PARAMETERS
1514
1515C<instantiate()> creates and returns an empty POE::Session object. It is
1516called with the CREATE_PARAMETERS in a hash reference just before
1517C<create()> processes them. Modifications to the CREATE_PARAMETERS will
1518affect how C<create()> initializes the new session.
1519
1520Subclasses may override C<instantiate()> to alter the underlying
1521session's structure. They may extend C<instantiate()> to add new
1522parameters to C<create()>.
1523
1524Any parameters not recognized by C<create()> must be removed from the
1525CREATE_PARAMETERS before C<instantiate()> returns. C<create()> will
1526L<croak|Carp/croak>
1527if it discovers unknown parameters.
1528
1529Be sure to return C<$self> from instantiate.
1530
1531 sub instantiate {
1532 my ($class, $create_params) = @_;
1533
1534 # Have the base class instantiate the new session.
1535 my $self = $class->SUPER::instantiate($create_parameters);
1536
1537 # Extend the parameters recognized by create().
1538 my $new_option = delete $create_parameters->{new_option};
1539 if (defined $new_option) {
1540 # ... customize $self here ...
1541 }
1542
1543 return $self;
1544 }

1566=head1 POE::Session's EVENTS
1567
1568Please do not define new events that begin with a leading underscore.
1569POE claims /^_/ events as its own.
1570
1571POE::Session only generates one event, _default. All other internal
1572POE events are generated by (and documented in) POE::Kernel.
1573
1574=head2 _default
1575
1576_default is the C<AUTOLOAD> of event handlers. If POE::Session can't
1577find a handler at dispatch time, it attempts to redirect the event to
1578_default's handler instead.
1579
1580If there's no _default handler, POE::Session will silently drop the
1581event unless the "default" option is set.
1582
1583To preserve the original information, the original event is slightly
1584changed before being redirected to the _default handler: The original
1585event parameters are moved to an array reference in ARG1, and the
1586original event name is passed to _default in ARG0.
1587
1588 sub handle_default {
1589 my ($event, $args) = @_[ARG0, ARG1];
1590 print(
1591 "Session ", $_[SESSION]->ID,
1592 " caught unhandled event $event with (@$args).\n"
1593 );
1594 }
1595
1596_default is quite flexible. It may be used for debugging, or to
1597handle dynamically generated event names without pre-defining their
1598handlers. In the latter sense, _default performs analogously to
1599Perl's C<AUTOLOAD>.
1600
1601_default may also be used as the default or "otherwise" clause of a
1602switch statement. Consider an input handler that throws events based
1603on a command name:
1604
1605 sub parse_command {
1606 my ($command, @parameters) = split /\s+/, $_[ARG0];
1607 $_[KERNEL]->post( "cmd_$command", @parameters );
1608 }
1609
1610A _default handler may be used to emit errors for unknown commands:
1611
1612 sub handle_default {
1613 my $event = $_[ARG0];
1614 return unless $event =~ /^cmd_(\S+)/;
1615 warn "Unknown command: $1\n";
1616 }
1617
1618The _default behavior is implemented in POE::Session, so it may be
1619different for other session types.

1647=head2 Beware circular references
1648
1649As you're probably aware, a circular reference is when a variable is
1650part of a reference chain that eventually refers back to itself. Perl
1651will not reclaim the memory involved in such a reference chain until
1652the chain is manually broken.
1653
1654Here a POE::Session is created that refers to itself via an external
1655scalar. The event handlers import $session via closures which are in
1656turn stored within $session. Even if this session stops, the circular
1657references will remain.
1658
1659 my $session;
1660 $session = POE::Session->create(
1661 inline_states => {
1662 _start => sub {
1663 $_[HEAP]->{todo} = [ qw( step1 step2 step2a ) ],
1664 $_[KERNEL]->post( $session, 'next' );
1665 },
1666 next => sub {
1667 my $next = shift @{ $_[HEAP]->{todo} };
1668 return unless $next;
1669 $_[KERNEL]->post( $session, $next );
1670 }
1671 # ....
1672 }
1673 );
1674
1675Reduced to its essence:
1676
1677 my %event_handlers;
1678 $event_handler{_start} = sub { \%event_handlers };
1679
1680Note also that a anonymous sub creates a closure on all lexical
1681variables in the scope it was defined in, even if it doesn't reference
1682them. $session is still being held in a circular reference here:
1683
1684 my $self = $package->new;
1685 my $session;
1686 $session = POE::Session->create(
1687 inline_state => {
1688 _start => sub { $self->_start( @_[ARG0..$#_] ) }
1689 }
1690 );
1691
1692To avoid this, a session may set an alias for itself. Other parts of
1693the program may then refer to it by alias. In this case, one needn't
1694keep track of the session themselves (POE::Kernel will do it anyway).
1695
1696 POE::Session->create(
1697 inline_states => {
1698 _start => sub {
1699 $_[HEAP]->{todo} = [ qw( step1 step2 step2a ) ],
1700 $_[KERNEL]->alias_set('step_doer');
1701 $_[KERNEL]->post( 'step_doer', 'next' );
1702 },
1703 next => sub {
1704 my $next = shift @{ $_[HEAP]->{todo} };
1705 return unless $next;
1706 $_[KERNEL]->post( 'step_doer', $next );
1707 }
1708 # ....
1709 }
1710 );
1711
1712Aliases aren't even needed in the previous example because the session
1713refers to itself. One could instead use POE::Kernel's yield() method
1714to post the event back to the current session:
1715
1716 next => sub {
1717 my $next = shift @{ $_[HEAP]->{todo} };
1718 return unless $next;
1719 $_[KERNEL]->yield( $next );
1720 }
1721
1722Or the L<$_[SESSION]> parameter passed to every event handler, but
1723yield() is more efficient.
1724
1725 next => sub {
1726 my $next = shift @{ $_[HEAP]->{todo} };
1727 return unless $next;
1728 $_[KERNEL]->post( $_[SESSION], $next );
1729 }
1730
1731Along the same lines as L<$_[SESSION]>, a session can respond back to
1732the sender of an event by posting to L<$_[SENDER]>. This is great for
1733responding to requests.
1734
1735If a program must hold onto some kind of dynamic session reference,
1736it's recommended to use the session's numeric ID rather than the
1737object itself. A session ID may be converted back into its object,
1738but post() accepts session IDs as well as objects and aliases:
1739
1740 my $session_id;
1741 $session_id = POE::Session->create(
1742 inline_states => {
1743 _start => sub {
1744 $_[HEAP]->{todo} = [ qw( step1 step2 step2a ) ],
1745 $_[KERNEL]->post( $session_id, 'next' );
1746 },
1747 # ....
1748 }
1749 )->ID;
