173=head1 NAME
174
175POE::Driver::SysRW - buffered, non-blocking I/O using sysread and syswrite
176
177=head1 SYNOPSIS
178
179L<POE::Driver/SYNOPSIS> illustrates how the interface works. This
180module is merely one implementation.
181
182=head1 DESCRIPTION
183
184This driver implements L<POE::Driver> using sysread and syswrite.
185
186=head1 PUBLIC METHODS
187
188POE::Driver::SysRW introduces some additional features not covered in
189the base interface.
190
191=head2 new [BlockSize => OCTETS]
192
193new() creates a new buffered I/O driver that uses sysread() to read
194data from a handle and syswrite() to flush data to that handle. The
195constructor accepts one optional named parameter, C<BlockSize>, which
196indicates the maximum number of OCTETS that will be read at one time.
197
198C<BlockSize> is 64 kilobytes (65536 octets) by default. Higher values
199may improve performance in streaming applications, but the trade-off
200is a lower event granularity and increased resident memory usage.
201
202Lower C<BlockSize> values reduce memory consumption somewhat with
203corresponding throughput penalties.
204
205 my $driver = POE::Driver::SysRW->new;
206
207 my $driver = POE::Driver::SysRW->new( BlockSize => $block_size );
208
209Drivers are commonly instantiated within POE::Wheel constructor calls:
210
211 $_[HEAP]{wheel} = POE::Wheel::ReadWrite->new(
212 InputHandle => \*STDIN,
213 OutputHandle => \*STDOUT,
214 Driver => POE::Driver::SysRW->new(),
215 Filter => POE::Filter::Line->new(),
216 );
217
218Applications almost always use POE::Driver::SysRW, so POE::Wheel
219objects almost always will create their own if no Driver is specified.
220
