2602=head1 DESCRIPTION
2603
2604POE::Kernel is the heart of POE. It provides the lowest-level
2605features: non-blocking multiplexed I/O, timers, and signal watchers
2606are the most significant. Everything else is built upon this
2607foundation.

2610several available POE::Loop interface modules. See CPAN for modules
2611in the POE::Loop namespace.

2617=head1 USING POE
2618
2619=head2 Literally Using POE
2620
2621POE.pm is little more than a class loader. It implements some magic
2622to cut down on the setup work.
2623

2632two modules are used by nearly all POE-based programs. So the above
2633example is actually the equivalent of:
2634
2635 use POE;

2641
2642POE::Kernel needs to know which event loop you want to use. This is

2644

2647
2648 use Tk; # or one of several others
2649 use POE::Kernel.

2656 use POE qw(Loop::Gtk);

2664=head2 Anatomy of a POE-Based Application
2665
2666Programs using POE work like any other. They load required modules,
2667perform some setup, run some code, and eventually exit. Halting
2668Problem notwithstanding.
2669
2670A POE-based application loads some modules, sets up one or more
2671sessions, runs the code in those sessions, and eventually exits.
2672
2673 use POE;
2674 POE::Session->create( ... map events to code here ... );
2675 POE::Kernel->run();
2676 exit;
2677
2678=head2 POE::Kernel singleton
2679
2680The POE::Kernel is a singleton object; there can be only one POE::Kernel
2681instance within a process. This allows many object methods to also be
2682package methods.
2683
2684=head2 Sessions
2685
2686POE implements isolated compartments called I<sessions>. Sessions play
2687the role of tasks or threads within POE. POE::Kernel acts as POE's
2688task scheduler, doling out timeslices to each session by invoking
2689callbacks within them.
2690
2691Callbacks are not preemptive. As long as one is running, no others
2692will be dispatched. This is known as I<cooperative> multitasking.
2693Each session must cooperate by returning to the central dispatching
2694kernel.
2695
2696Cooperative multitasking vastly simplifies data sharing, since no two
2697pieces of code may alter data at once.
2698
2699A session may also take exclusive control of a program's time, if
2700necessary, by simply not returning in a timely fashion. It's even
2701possible to write completely blocking programs that use POE as a state
2702machine rather than a cooperative dispatcher.
2703
2704Every POE-based application needs at least one session. Code cannot
2705run I<within POE> without being a part of some session. Likewise, a
2706threaded program always has a "thread zero".
2707
2708Sessions in POE::Kernel should not be confused with
2709L<POE::Session|POE::Session> even though the two are inextricably
2710associated. POE::Session adapts POE::Kernel's dispatcher to a
2711particular calling convention. Other POE::Session classes exist on
2712the CPAN. Some radically alter the way event handlers are called.
2713L<http://search.cpan.org/search?query=poe+session>.
2714
2715=head2 Resources
2716
2717Resources are events and things which may create new events, such as
2718timers, I/O watchers, and even other sessions.
2719
2720POE::Kernel tracks resources on behalf of its active sessions. It
2721generates events corresponding to these resources' activity, notifying
2722sessions when it's time to do things.
2723
2724The conversation goes something like this:
2725
2726 Session: Be a dear, Kernel, and let me know when someone clicks on
2727 this widget. Thanks so much!
2728
2729 [TIME PASSES] [SFX: MOUSE CLICK]
2730
2731 Kernel: Right, then. Someone's clicked on your widget.
2732 Here you go.
2733
2734Furthermore, since the Kernel keeps track of everything sessions do,
2735it knows when a session has run out of tasks to perform. When this
2736happens, the Kernel emits a C<_stop> event at the dead session so it
2737can clean up and shutdown.
2738
2739 Kernel: Please switch off the lights and lock up; it's time to go.
2740
2741Likewise, if a session stops on its own and there still are opened
2742resource watchers, the Kernel knows about them and cleans them up on
2743the session's behalf. POE excels at long-running services because it
2744so meticulously tracks and cleans up resources.
2745
2746POE::Resources and the POE::Resource classes implement each kind of
2747resource, which are summarized here and covered in greater detail
2748later.
2749
2750=over 2
2751
2752=item Events.
2753
2754An event is a message to a sessions. Posting an event keeps both the
2755sender and the receiver alive until after the event has been
2756dispatched. This is only guaranteed if both the sender and receiver
2757are in the same process. Inter-Kernel message passing add-ons may
2758have other guarantees. Please see their documentation for details.
2759
2760The rationale is that the event is in play, so the receiver must
2761remain active for it to be dispatched. The sender remains alive in
2762case the receiver would like to send back a response.
2763
2764Posted events cannot be preemptively canceled. They tend to be
2765short-lived in practice, so this generally isn't an issue.
2766
2767=item Timers.
2768
2769Timers allow an application to send a message to the future. Once set,
2770a timer will keep the destination session active until it goes off and
2771the resulting event is dispatched.
2772
2773=item Aliases.
2774
2775Session aliases are an application-controlled way of addressing a
2776session. Aliases act as passive event watchers. As long as a session
2777has an alias, some other session may send events to that session by
2778that name. Aliases keep sessions alive as long as a process has
2779active sessions.
2780
2781If the only sessions remaining are being kept alive solely by their
2782aliases, POE::Kernel will send them a terminal L</IDLE> signal. In
2783most cases this will terminate the remaining sessions and allow the
2784program to exit. If the sessions remain in memory without waking up
2785on the C<IDLE> signal, POE::Kernel sends them a non-maskable L</ZOMBIE>
2786signal. They are then forcibly removed, and the program will finally
2787exit.
2788
2789=item I/O watchers.
2790
2791A session will remain active as long as a session is paying attention
2792to some external data source or sink. See
2793L<select_read|"select_read FILE_HANDLE [, EVENT_NAME [, ADDITIONAL_PARAMETERS] ]">
2794and
2795L<select_write|"select_write FILE_HANDLE [, EVENT_NAME [, ADDITIONAL_PARAMETERS] ]">.
2796
2797=item Child sessions.
2798
2799A session acting as a parent of one or more other sessions will remain
2800active until all the child sessions stop. This may be bypassed by
2801detaching the children from the parent.
2802
2803=item Child processes.
2804
2805Child process are watched by sig_child(). The sig_child() watcher
2806will keep the watching session active until the child process has been
2807reaped by POE::Kernel and the resulting event has been dispatched.
2808
2809All other signal watchers, including using L</sig> to watch for
2810C<CHLD>, do not keep their sessions active. If you need a session to
2811remain active when it's only watching for signals, have it set an
2812alias or one of its own public reference counters.
2813
2814=item Public reference counters.
2815
2816A session will remain active as long as it has one or more nonzero
2817public (or external) reference counter.
2818
2819=back

2821=head2 Session Lifespans
2822
2823"Session" as a term is somewhat overloaded. There are two related
2824concepts that share the name. First there is the class POE::Session,
2825and objects created with it or related classes. Second there is a
2826data structure within POE::Kernel that tracks the POE::Session objects
2827in play and the various resources owned by each.
2828
2829The way POE's garbage collector works is that a session object gives
2830itself to POE::Kernel at creation time. The Kernel then holds onto
2831that object as long as resources exist that require the session to
2832remain alive. When all of these resources are destroyed or released,
2833the session object has nothing left to trigger activity. POE::Kernel
2834notifies the object it's through, and cleans up its internal session
2835context. The session object is released, and self-destructs in the
2836normal Perlish fashion.
2837
2838Sessions may be stopped even if they have active resources. For
2839example, a session may fail to handle a terminal signal. In this
2840case, POE::Kernel forces the session to stop, and all resources
2841associated with the session are preemptively released.

2846application to another. An event consists of the event's name,
2847optional event-specific parameters and OOB information. An event may
2848be sent from the kernel, from a wheel or from a session.
2849
2850An application creates an event with L</post>, L</yield>, L</call> or
2851even L</signal>. POE::Kernel creates events in response external
2852stimulus (signals, select, etc).

2857
2858An event is handled by a function called an I<event handler>, which is
2859some code that is designated to be called when a particular event is
2860dispatched. See L</Event Handler Management> and L<POE::Session>.

2866(i.e. via post()). Synchronous events, invoked with call(), are
2867handled in the same context that call() was called.

2870"POE" namespace. POE::Kernel will stringify these references to
2871prevent timing issues with certain objects' destruction. For example,
2872this error handler would cause errors because a deleted wheel would
2873not be destructed when one might think:

2881returned implicitly by handle_error().

2883=head2 Using POE with Other Event Loops
2884
2885POE::Kernel supports any number of event loops. Two are included in
2886the base distribution. Historically, POE included other loops but they
2887were moved into a separate distribution. You can find them and other
2888loops on the CPAN.

2891being used. Since most graphical toolkits include some form of event
2892loop, back-end code should be portable to all of them.

2895other software. The common underlying event loop drives both the
2896application and POE. For example, by using POE::Loop::Glib, one can
2897embed POE into Vim, irssi, and so on. Application scripts can then
2898take advantage of POE::Component::Client::HTTP (and everything else)
2899to do large-scale work without blocking the rest of the program.
2900
2901Because this is Perl, there are multiple ways to load an alternate
2902event loop. The simplest way is to load the event loop before loading
2903POE::Kernel.
2904
2905 use Gtk;
2906 use POE;
2907
2908Remember that POE loads POE::Kernel internally.
2909
2910POE::Kernel examines the modules loaded before it and detects that
2911L<Gtk> has been loaded. If L<POE::Loop::Gtk|POE::Loop::Gtk> is
2912available, POE loads and hooks it into POE::Kernel automatically.
2913
2914It's less mysterious to load the appropriate L<POE::Loop|POE::Loop>
2915class directly. Their names follow the format
2916C<POE::Loop::$loop_module_name>, where C<$loop_module_name> is the
2917name of the event loop module after each C<::> has been substituted
2918with an underscore. It can be abbreviated using POE's loader magic.
2919
2920 use POE qw( Loop::Event_Lib );
2921
2922POE also recognizes XS loops, they reside in the
2923C<POE::XS::Loop::$loop_module_name> namespace. Using them may give
2924you a performance improvement on your platform, as the eventloop
2925are some of the hottest code in the system. As always, benchmark
2926your application against various loops to see which one is best for
2927your workload and platform.
2928
2929 use POE qw( XS::Loop::EPoll );
2930
2931Please don't load the loop modules directly, because POE will not have
2932a chance to initialize it's internal structures yet. Code written like
2933this will throw errors on startup. It might look like a bug in POE, but
2934it's just the way POE is designed.
2935
2936 use POE::Loop::IO_Poll;
2937 use POE;
2938
2939POE::Kernel also supports configuration directives on its own C<use>
2940line. A loop explicitly specified this way will override the search
2941logic.
2942
2943 use POE::Kernel { loop => "Glib" };
2944
2945Finally, one may specify the loop class by setting the POE::Loop or
2946POE::XS:Loop class name in the POE_EVENT_LOOP environment variable.
2947This mechanism was added for tests that need to specify the loop from
2948a distance.
2949
2950 BEGIN { $ENV{POE_EVENT_LOOP} = "POE::XS::Loop::Poll" }
2951 use POE;
2952
2953Of course this may also be set from your shell:
2954
2955 % export POE_EVENT_LOOP='POE::XS::Loop::Poll'
2956 % make test

3056
3057=head3 ID
3058
3059ID() returns the kernel's unique identifier. Every POE::Kernel
3060instance is assigned a (hopefully) globally unique ID at birth.
3061
3062 % perl -wl -MPOE -e 'print $poe_kernel->ID'
3063 poerbook.local-46c89ad800000e21
3064
3065While the IDs are made globally unique by including hostname, time and
3066PID, they should be considered an opaque but printable string. That
3067is, your code should not depend on the current format.
3068
3069=head3 run
3070
3071run() runs POE::Kernel's event dispatcher. It will not return until
3072all sessions have ended. run() is a class method so a POE::Kernel
3073reference is not needed to start a program's execution.
3074
3075 use POE;
3076 POE::Session->create( ... ); # one or more
3077 POE::Kernel->run(); # set them all running
3078 exit;
3079
3080POE implements the Reactor pattern at its core. Events are dispatched
3081to functions and methods through callbacks. The code behind run()
3082waits for and dispatches events.
3083
3084run() will not return until every session has ended. This includes
3085sessions that were created while run() was running.
3086
3087POE::Kernel will print a strong message if a program creates sessions
3088but fails to call run(). Prior to this warning, we received tons of
3089bug reports along the lines of "my POE program isn't doing anything".
3090It turned out that people forgot to start an event dispatcher, so
3091events were never dispatched.
3092
3093If the lack of a run() call is deliberate, perhaps because some other
3094event loop already has control, you can avoid the message by calling
3095it before creating a session. run() at that point will initialize POE
3096and return immediately. POE::Kernel will be satisfied that run() was
3097called, although POE will not have actually taken control of the event
3098loop.
3099
3100 use POE;
3101 POE::Kernel->run(); # silence the warning
3102 POE::Session->create( ... );
3103 exit;
3104
3105Note, however, that this varies from one event loop to another. If a
3106particular POE::Loop implementation doesn't support it, that's
3107probably a bug. Please file a bug report with the owner of the
3108relevant POE::Loop module.

3110=head3 run_one_timeslice
3111
3112run_one_timeslice() dispatches any events that are due to be
3113delivered. These events include timers that are due, asynchronous
3114messages that need to be delivered, signals that require handling, and
3115notifications for files with pending I/O. Do not rely too much on
3116event ordering. run_one_timeslice() is defined by the underlying
3117event loop, and its timing may vary.
3118
3119run() is implemented similar to
3120
3121 run_one_timeslice() while $session_count > 0;
3122
3123run_one_timeslice() can be used to keep running POE::Kernel's
3124dispatcher while emulating blocking behavior. The pattern is
3125implemented with a flag that is set when some asynchronous event
3126occurs. A loop calls run_one_timeslice() until that flag is set. For
3127example:
3128
3129 my $done = 0;
3130
3131 sub handle_some_event {
3132 $done = 1;
3133 }
3134
3135 $kernel->run_one_timeslice() while not $done;
3136
3137Do be careful. The above example will spin if POE::Kernel is done but
3138$done is never set. The loop will never be done, even though there's
3139nothing left that will set $done.
3140
3141=head3 run_while SCALAR_REF
3142
3143run_while() is an B<experimental> version of run_one_timeslice() that
3144will only return when there are no more active sessions, or the value
3145of the referenced scalar becomes false.
3146
3147Here's a version of the run_one_timeslice() example using run_while()
3148instead:
3149
3150 my $job_count = 3;
3151
3152 sub handle_some_event {
3153 $job_count--;
3154 }
3155
3156 $kernel->run_while(\$job_count);

3158=head3 has_forked
3159
3160 my $pid = fork();
3161 die "Unable to fork" unless defined $pid;
3162 unless( $pid ) {
3163 $poe_kernel->has_forked;
3164 }
3165
3166Inform the kernel that it is now running in a new process. This allows the
3167kernel to reset some internal data to adjust to the new situation.
3168
3169has_forked() must be called in the child process if you wish to run the same
3170kernel. However, if you want the child process to have new kernel, you must
3171call L</stop> instead.
3172

3174=head3 stop
3175
3176stop() causes POE::Kernel->run() to return early. It does this by
3177emptying the event queue, freeing all used resources, and stopping
3178every active session. stop() is not meant to be used lightly.
3179Proceed with caution.
3180
3181Caveats:
3182
3183The session that calls stop() will not be fully DESTROYed until it
3184returns. Invoking an event handler in the session requires a
3185reference to that session, and weak references are prohibited in POE
3186for backward compatibility reasons, so it makes sense that the last
3187session won't be garbage collected right away.
3188
3189Sessions are not notified about their destruction. If anything relies
3190on _stop being delivered, it will break and/or leak memory.
3191
3192stop() is still considered experimental. It was added to improve fork()
3193support for L<POE::Wheel::Run|POE::Wheel::Run>. If it proves unfixably
3194problematic, it will be removed without much notice.
3195
3196stop() is advanced magic. Programmers who think they need it are
3197invited to become familiar with its source.
3198
3199See L<POE::Wheel::Run/Nested POE Kernel> for an example of how to use this
3200facility.

3202=head2 Asynchronous Messages (FIFO Events)
3203
3204Asynchronous messages are events that are dispatched in the order in
3205which they were enqueued (the first one in is the first one out,
3206otherwise known as first-in/first-out, or FIFO order). These methods
3207enqueue new messages for delivery. The act of enqueuing a message
3208keeps the sender alive at least until the message is delivered.
3209
3210=head3 post DESTINATION, EVENT_NAME [, PARAMETER_LIST]
3211
3212post() enqueues a message to be dispatched to a particular DESTINATION
3213session. The message will be handled by the code associated with
3214EVENT_NAME. If a PARAMETER_LIST is included, its values will also be
3215passed along.
3216
3217 POE::Session->create(
3218 inline_states => {
3219 _start => sub {
3220 $_[KERNEL]->post( $_[SESSION], "event_name", 0 );
3221 },
3222 event_name => sub {
3223 print "$_[ARG0]\n";
3224 $_[KERNEL]->post( $_[SESSION], "event_name", $_[ARG0] + 1 );
3225 },
3226 }
3227 );
3228
3229post() returns a Boolean value indicating whether the message was
3230successfully enqueued. If post() returns false, $! is set to explain
3231the failure:
3232
3233ESRCH ("No such process") - The DESTINATION session did not exist at
3234the time post() was called.
3235
3236=head3 yield EVENT_NAME [, PARAMETER_LIST]
3237
3238yield() is a shortcut for post() where the destination session is the
3239same as the sender. This example is equivalent to the one for post():
3240
3241 POE::Session->create(
3242 inline_states => {
3243 _start => sub {
3244 $_[KERNEL]->yield( "event_name", 0 );
3245 },
3246 event_name => sub {
3247 print "$_[ARG0]\n";
3248 $_[KERNEL]->yield( "event_name", $_[ARG0] + 1 );
3249 },
3250 }
3251 );
3252
3253As with post(), yield() returns right away, and the enqueued
3254EVENT_NAME is dispatched later. This may be confusing if you're
3255already familiar with threading.
3256
3257yield() should always succeed, so it does not return a meaningful
3258value.
3259
3260=head2 Synchronous Messages
3261
3262It is sometimes necessary for code to be invoked right away. For
3263example, some resources must be serviced right away, or they'll
3264faithfully continue reporting their readiness. These reports would
3265appear as a stream of duplicate events. Synchronous events can also
3266prevent data from going stale between the time an event is enqueued
3267and the time it's delivered.
3268
3269Synchronous event handlers preempt POE's event queue, so they should
3270perform simple tasks of limited duration. Synchronous events that
3271need to do more than just service a resource should pass the
3272resource's information to an asynchronous handler. Otherwise
3273synchronous operations will occur out of order in relation to
3274asynchronous events. It's very easy to have race conditions or break
3275causality this way, so try to avoid it unless you're okay with the
3276consequences.
3277
3278POE provides these ways to call message handlers right away.
3279
3280=head3 call DESTINATION, EVENT_NAME [, PARAMETER_LIST]
3281
3282call()'s semantics are nearly identical to post()'s. call() invokes a
3283DESTINATION's handler associated with an EVENT_NAME. An optional
3284PARAMETER_LIST will be passed along to the message's handler. The
3285difference, however, is that the handler will be invoked immediately,
3286even before call() returns.
3287
3288call() returns the value returned by the EVENT_NAME handler. It can
3289do this because the handler is invoked before call() returns. call()
3290can therefore be used as an accessor, although there are better ways
3291to accomplish simple accessor behavior.
3292
3293 POE::Session->create(
3294 inline_states => {
3295 _start => sub {
3296 print "Got: ", $_[KERNEL]->call($_[SESSION], "do_now"), "\n";
3297 },
3298 do_now => sub {
3299 return "some value";
3300 }
3301 }
3302 );
3303
3304The L<POE::Wheel|POE::Wheel> classes uses call() to synchronously deliver I/O
3305notifications. This avoids a host of race conditions.
3306
3307call() may fail in the same way and for the same reasons as post().
3308On failure, $! is set to some nonzero value indicating why. Since
3309call() may return undef as a matter of course, it's recommended that
3310$! be checked for the error condition as well as the explanation.
3311
3312ESRCH ("No such process") - The DESTINATION session did not exist at
3313the time post() was called.

3315=head2 Timer Events (Delayed Messages)
3316
3317It's often useful to wait for a certain time or until a certain amount
3318of time has passed. POE supports this with events that are deferred
3319until either an absolute time ("alarms") or until a certain duration
3320of time has elapsed ("delays").
3321
3322Timer interfaces are further divided into two groups. One group identifies
3323timers by the names of their associated events. Another group identifies
3324timers by a unique identifier returned by the timer constructors.
3325Technically, the two are both name-based, but the "identifier-based" timers
3326provide a second, more specific handle to identify individual timers.
3327
3328Timers may only be set up for the current session. This design was
3329modeled after alarm() and SIGALRM, which only affect the current UNIX
3330process. Each session has a separate namespace for timer names.
3331Timer methods called in one session cannot affect the timers in
3332another. As you may have noticed, quite a lot of POE's API is
3333designed to prevent sessions from interfering with each other.
3334
3335The best way to simulate deferred inter-session messages is to send an
3336immediate message that causes the destination to set a timer. The
3337destination's timer then defers the action requested of it. This way
3338is preferred because the time spent communicating the request between
3339sessions may not be trivial, especially if the sessions are separated
3340by a network. The destination can determine how much time remains on
3341the requested timer and adjust its wait time accordingly.
3342
3343=head3 Using Time::HiRes
3344
3345POE::Kernel timers support sub-second accuracy, but don't expect too
3346much here. Perl is not the right language for realtime programming.
3347
3348Subsecond accuracy is supported through the use of select() timeouts
3349and other event-loop features. For increased accuracy, POE::Kernel
3350uses L<Time::HiRes|Time::HiRes>'s time() internally, if it's available.
3351
3352You can disable POE's use of Time::HiRes by defining a constant in the
3353POE::Kernel namespace. This must be done before POE::Kernel is
3354loaded, so that the compiler can use it.
3355
3356 BEGIN {
3357 package POE::Kernel;
3358 use constant USE_TIME_HIRES => 0;
3359 }
3360 use POE;
3361
3362Or the old-fashioned (and more concise) "constant subroutine" method.
3363This doesn't need the C<BEGIN{}> block since subroutine definitions are
3364done at compile time.
3365
3366 sub POE::Kernel::USE_TIME_HIRES () { 0 }
3367 use POE;
3368
3369=head3 Name-Based Timers
3370
3371Name-based timers are identified by the event names used to set them.
3372It is possible for different sessions to use the same timer event names,
3373since each session is a separate compartment with its own timer namespace.
3374It is possible for a session to have multiple timers for a given event,
3375but results may be surprising. Be careful to use the right timer methods.
3376
3377The name-based timer methods are alarm(), alarm_add(), delay(), and
3378delay_add().
3379
3380=head4 alarm EVENT_NAME [, EPOCH_TIME [, PARAMETER_LIST] ]
3381
3382alarm() clears all existing timers in the current session with the
3383same EVENT_NAME. It then sets a new timer, named EVENT_NAME, that
3384will fire EVENT_NAME at the current session when EPOCH_TIME has been
3385reached. An optional PARAMETER_LIST may be passed along to the
3386timer's handler.
3387
3388Omitting the EPOCH_TIME and subsequent parameters causes alarm() to
3389clear the EVENT_NAME timers in the current session without setting a
3390new one.
3391
3392EPOCH_TIME is the UNIX epoch time. You know, seconds since midnight,
33931970-01-01. "Now" is whatever time() returns, either the built-in or
3394L<Time::HiRes|Time::HiRes> version. POE will use Time::HiRes if it's
3395available.
3396
3397POE supports fractional seconds, but accuracy falls off steeply after
33981/100 second. Mileage will vary depending on your CPU speed and your
3399OS time resolution.
3400
3401Be sure to use Time::HiRes::time() rather than Perl's built-in time()
3402if sub-second accuracy matters at all. The built-in time() returns
3403floor(Time::HiRes::time()), which is nearly always some fraction of a
3404second in the past. For example the high-resolution time might be
34051200941422.89996. At that same instant, time() would be 1200941422.
3406An alarm for time() + 0.5 would be 0.39996 seconds in the past, so it
3407would be dispatched immediately (if not sooner).
3408
3409POE's event queue is time-ordered, so a timer due before time() will
3410be delivered ahead of other events but not before timers with even
3411earlier due times. Therefore an alarm() with an EPOCH_TIME before
3412time() jumps ahead of the queue.
3413
3414All timers are implemented identically internally, regardless of how
3415they are set. alarm() will therefore blithely clear timers set by
3416other means.
3417
3418 POE::Session->create(
3419 inline_states => {
3420 _start => sub {
3421 $_[KERNEL]->alarm( tick => time() + 1, 0 );
3422 },
3423 tick => sub {
3424 print "tick $_[ARG0]\n";
3425 $_[KERNEL]->alarm( tock => time() + 1, $_[ARG0] + 1 );
3426 },
3427 tock => sub {
3428 print "tock $_[ARG0]\n";
3429 $_[KERNEL]->alarm( tick => time() + 1, $_[ARG0] + 1 );
3430 },
3431 }
3432 );
3433
3434alarm() returns 0 on success or a true value on failure. Usually
3435EINVAL to signal an invalid parameter, such as an undefined
3436EVENT_NAME.
3437
3438=head4 alarm_add EVENT_NAME, EPOCH_TIME [, PARAMETER_LIST]
3439
3440alarm_add() is used to add a new alarm timer named EVENT_NAME without
3441clearing existing timers. EPOCH_TIME is a required parameter.
3442Otherwise the semantics are identical to alarm().
3443
3444A program may use alarm_add() without first using alarm().
3445
3446 POE::Session->create(
3447 inline_states => {
3448 _start => sub {
3449 $_[KERNEL]->alarm_add( tick => time() + 1.0, 1_000_000 );
3450 $_[KERNEL]->alarm_add( tick => time() + 1.5, 2_000_000 );
3451 },
3452 tick => sub {
3453 print "tick $_[ARG0]\n";
3454 $_[KERNEL]->alarm_add( tock => time() + 1, $_[ARG0] + 1 );
3455 },
3456 tock => sub {
3457 print "tock $_[ARG0]\n";
3458 $_[KERNEL]->alarm_add( tick => time() + 1, $_[ARG0] + 1 );
3459 },
3460 }
3461 );
3462
3463alarm_add() returns 0 on success or EINVAL if EVENT_NAME or EPOCH_TIME
3464is undefined.

3466=head4 delay EVENT_NAME [, DURATION_SECONDS [, PARAMETER_LIST] ]
3467
3468delay() clears all existing timers in the current session with the
3469same EVENT_NAME. It then sets a new timer, named EVENT_NAME, that
3470will fire EVENT_NAME at the current session when DURATION_SECONDS have
3471elapsed from "now". An optional PARAMETER_LIST may be passed along to
3472the timer's handler.
3473
3474Omitting the DURATION_SECONDS and subsequent parameters causes delay()
3475to clear the EVENT_NAME timers in the current session without setting
3476a new one.
3477
3478DURATION_SECONDS may be or include fractional seconds. As with all of
3479POE's timers, accuracy falls off steeply after 1/100 second. Mileage
3480will vary depending on your CPU speed and your OS time resolution.
3481
3482POE's event queue is time-ordered, so a timer due before time() will
3483be delivered ahead of other events but not before timers with even
3484earlier due times. Therefore a delay () with a zero or negative
3485DURATION_SECONDS jumps ahead of the queue.
3486
3487delay() may be considered a shorthand form of alarm(), but there are
3488subtle differences in timing issues. This code is roughly equivalent
3489to the alarm() example.
3490
3491 POE::Session->create(
3492 inline_states => {
3493 _start => sub {
3494 $_[KERNEL]->delay( tick => 1, 0 );
3495 },
3496 tick => sub {
3497 print "tick $_[ARG0]\n";
3498 $_[KERNEL]->delay( tock => 1, $_[ARG0] + 1 );
3499 },
3500 tock => sub {
3501 print "tock $_[ARG0]\n";
3502 $_[KERNEL]->delay( tick => 1, $_[ARG0] + 1 );
3503 },
3504 }
3505 );
3506
3507delay() returns 0 on success or a reason for failure: EINVAL if
3508EVENT_NAME is undefined.
3509
3510=head4 delay_add EVENT_NAME, DURATION_SECONDS [, PARAMETER_LIST]
3511
3512delay_add() is used to add a new delay timer named EVENT_NAME without
3513clearing existing timers. DURATION_SECONDS is a required parameter.
3514Otherwise the semantics are identical to delay().
3515
3516A program may use delay_add() without first using delay().
3517
3518 POE::Session->create(
3519 inline_states => {
3520 _start => sub {
3521 $_[KERNEL]->delay_add( tick => 1.0, 1_000_000 );
3522 $_[KERNEL]->delay_add( tick => 1.5, 2_000_000 );
3523 },
3524 tick => sub {
3525 print "tick $_[ARG0]\n";
3526 $_[KERNEL]->delay_add( tock => 1, $_[ARG0] + 1 );
3527 },
3528 tock => sub {
3529 print "tock $_[ARG0]\n";
3530 $_[KERNEL]->delay_add( tick => 1, $_[ARG0] + 1 );
3531 },
3532 }

3535delay_add() returns 0 on success or EINVAL if EVENT_NAME or EPOCH_TIME
3536is undefined.

3538=head3 Identifier-Based Timers
3539
3540A second way to manage timers is through identifiers. Setting an
3541alarm or delay with the "identifier" methods allows a program to
3542manipulate several timers with the same name in the same session. As
3543covered in alarm() and delay() however, it's possible to mix named and
3544identified timer calls, but the consequences may not always be
3545expected.
3546
3547=head4 alarm_set EVENT_NAME, EPOCH_TIME [, PARAMETER_LIST]
3548
3549alarm_set() sets an alarm, returning a unique identifier that can be
3550used to adjust or remove the alarm later. Unlike alarm(), it does not
3551first clear existing timers with the same EVENT_NAME. Otherwise the
3552semantics are identical to alarm().
3553
3554 POE::Session->create(
3555 inline_states => {
3556 _start => sub {
3557 $_[HEAP]{alarm_id} = $_[KERNEL]->alarm_set(
3558 party => time() + 1999
3559 );
3560 $_[KERNEL]->delay(raid => 1);
3561 },
3562 raid => sub {
3563 $_[KERNEL]->alarm_remove( delete $_[HEAP]{alarm_id} );
3564 },
3565 }
3566 );
3567
3568alarm_set() returns false if it fails and sets $! with the
3569explanation. $! will be EINVAL if EVENT_NAME or TIME is undefined.
3570
3571=head4 alarm_adjust ALARM_ID, DELTA_SECONDS
3572
3573alarm_adjust() adjusts an existing timer's due time by DELTA_SECONDS,
3574which may be positive or negative. It may even be zero, but that's
3575not as useful. On success, it returns the timer's new due time since
3576the start of the UNIX epoch.
3577
3578It's possible to alarm_adjust() timers created by delay_set() as well
3579as alarm_set().
3580
3581This example moves an alarm's due time ten seconds earlier.
3582
3583 use POSIX qw(strftime);
3584
3585 POE::Session->create(
3586 inline_states => {
3587 _start => sub {
3588 $_[HEAP]{alarm_id} = $_[KERNEL]->alarm_set(
3589 party => time() + 1999
3590 );
3591 $_[KERNEL]->delay(postpone => 1);
3592 },
3593 postpone => sub {
3594 my $new_time = $_[KERNEL]->alarm_adjust(
3595 $_[HEAP]{alarm_id}, -10
3596 );
3597 print(
3598 "Now we're gonna party like it's ",
3599 strftime("%F %T", gmtime($new_time)), "\n"
3600 );
3601 },
3602 }
3603 );
3604
3605alarm_adjust() returns Boolean false if it fails, setting $! to the
3606reason why. $! may be EINVAL if ALARM_ID or DELTA_SECONDS are
3607undefined. It may be ESRCH if ALARM_ID no longer refers to a pending
3608timer. $! may also contain EPERM if ALARM_ID is valid but belongs to
3609a different session.

3611=head4 alarm_remove ALARM_ID
3612
3613alarm_remove() removes the alarm identified by ALARM_ID. ALARM_ID
3614comes from a previous alarm_set() or delay_set() call.
3615
3616Upon success, alarm_remove() returns something true based on its
3617context. In a list context, it returns three things: The removed
3618alarm's event name, the UNIX time it was due to go off, and a
3619reference to the PARAMETER_LIST (if any) assigned to the timer when it
3620was created. If necessary, the timer can be re-set with this
3621information.
3622
3623 POE::Session->create(
3624 inline_states => {
3625 _start => sub {
3626 $_[HEAP]{alarm_id} = $_[KERNEL]->alarm_set(
3627 party => time() + 1999
3628 );
3629 $_[KERNEL]->delay(raid => 1);
3630 },
3631 raid => sub {
3632 my ($name, $time, $param) = $_[KERNEL]->alarm_remove(
3633 $_[HEAP]{alarm_id}
3634 );
3635 print(
3636 "Removed alarm for event $name due at $time with @$param\n"
3637 );
3638
3639 # Or reset it, if you'd like. Possibly after modification.
3640 $_[KERNEL]->alarm_set($name, $time, @$param);
3641 },
3642 }
3643 );
3644
3645In a scalar context, it returns a reference to a list of the three
3646things above.
3647
3648 # Remove and reset an alarm.
3649 my $alarm_info = $_[KERNEL]->alarm_remove( $alarm_id );
3650 my $new_id = $_[KERNEL]->alarm_set(
3651 $alarm_info[0], $alarm_info[1], @{$alarm_info[2]}
3652 );
3653
3654Upon failure, however, alarm_remove() returns a Boolean false value
3655and sets $! with the reason why the call failed:
3656
3657EINVAL ("Invalid argument") indicates a problem with one or more
3658parameters, usually an undefined ALARM_ID.
3659
3660ESRCH ("No such process") indicates that ALARM_ID did not refer to a
3661pending alarm.
3662
3663EPERM ("Operation not permitted"). A session cannot remove an alarm
3664it does not own.

3666=head4 alarm_remove_all

3669session, regardless of creation method or type. This method takes no
3670arguments. It returns information about the alarms that were removed,
3671either as a list of alarms or a list reference depending whether
3672alarm_remove_all() is called in scalar or list context.
3673
3674Each removed alarm's information is identical to the format explained
3675in alarm_remove().
3676
3677 sub some_event_handler {

3746=head2 Session Identifiers (IDs and Aliases)
3747
3748A session may be referred to by its object references (either blessed
3749or stringified), a session ID, or one or more symbolic names we call
3750aliases.
3751
3752Every session is represented by an object, so session references are
3753fairly straightforward. POE::Kernel may reference these objects. For
3754instance, post() may use $_[SENDER] as a destination:
3755
3756 POE::Session->create(
3757 inline_states => {
3758 _start => sub { $_[KERNEL]->alias_set("echoer") },
3759 ping => sub {
3760 $_[KERNEL]->post( $_[SENDER], "pong", @_[ARG0..$#_] );
3761 }
3762 }
3763 );
3764
3765POE also recognized stringified Session objects for convenience and as
3766a form of weak reference. Here $_[SENDER] is wrapped in quotes to
3767stringify it:
3768
3769 POE::Session->create(
3770 inline_states => {
3771 _start => sub { $_[KERNEL]->alias_set("echoer") },
3772 ping => sub {
3773 $_[KERNEL]->post( "$_[SENDER]", "pong", @_[ARG0..$#_] );
3774 }
3775 }
3776 );
3777
3778Every session is assigned a unique ID at creation time. No two active
3779sessions will have the same ID, but IDs may be reused over time. The
3780combination of a kernel ID and a session ID should be sufficient as a
3781global unique identifier.
3782
3783 POE::Session->create(
3784 inline_states => {
3785 _start => sub { $_[KERNEL]->alias_set("echoer") },
3786 ping => sub {
3787 $_[KERNEL]->delay(
3788 pong_later => rand(5), $_[SENDER]->ID, @_[ARG0..$#_]
3789 );
3790 },
3791 pong_later => sub {
3792 $_[KERNEL]->post( $_[ARG0], "pong", @_[ARG1..$#_] );
3793 }
3794 }
3795 );
3796
3797Kernels also maintain a global session namespace or dictionary from which
3798may be used to map a symbolic aliases to a session. Once an alias is mapping
3799has been created, that alias may be used to refer to the session wherever a
3800session may be specified.
3801
3802In the previous examples, each echoer service has set an "echoer"
3803alias. Another session can post a ping request to the echoer session
3804by using that alias rather than a session object or ID. For example:
3805
3806 POE::Session->create(
3807 inline_states => {
3808 _start => sub { $_[KERNEL]->post(echoer => ping => "whee!" ) },
3809 pong => sub { print "@_[ARG0..$#_]\n" }
3810 }
3811 );
3812
3813A session with an alias will not stop until all other activity has stopped.
3814Aliases are treated as a kind of event watcher. Events come from active
3815sessions. Aliases therefore become useless when there are no active
3816sessions left. Rather than leaving the program running in a "zombie" state,
3817POE detects this deadlock condition and triggers a cleanup. See
3818L</Signal Classes> for more information.

3820=head3 alias_set ALIAS
3821
3822alias_set() maps an ALIAS in POE::Kernel's dictionary to the
3823current session. The ALIAS may then be used nearly everywhere a session
3824reference, stringified reference, or ID is expected.
3825
3826Sessions may have more than one alias. Each alias must be defined in
3827a separate alias_set() call. A single alias may not refer to more
3828than one session.
3829
3830Multiple alias examples are above.
3831
3832alias_set() returns 0 on success, or a nonzero failure indicator:
3833EEXIST ("File exists") indicates that the alias is already assigned to
3834to a different session.
3835
3836=head3 alias_remove ALIAS
3837
3838alias_remove() removes an ALIAS for the current session from
3839POE::Kernel's dictionary. The ALIAS will no longer refer to the
3840current session. This does not negatively affect events already
3841posted to POE's queue. Alias resolution occurs at post() time, not at
3842delivery time.
3843
3844 POE::Session->create(
3845 inline_states => {
3846 _start => sub {
3847 $_[KERNEL]->alias_set("short_window");
3848 $_[KERNEL]->delay(close_window => 1);
3849 },
3850 close_window => {
3851 $_[KERNEL]->alias_remove("short_window");
3852 }
3853 }
3854 );
3855
3856alias_remove() returns 0 on success or a nonzero failure code: ESRCH
3857("No such process") indicates that the ALIAS is not currently in
3858POE::Kernel's dictionary. EPERM ("Operation not permitted") means
3859that the current session may not remove the ALIAS because it is in use
3860by some other session.

3862=head3 alias_resolve ALIAS
3863
3864alias_resolve() returns a session reference corresponding to a given
3865ALIAS. Actually, the ALIAS may be a stringified session reference, a
3866session ID, or an alias previously registered by alias_set().
3867
3868One use for alias_resolve() is to detect whether another session has
3869gone away:
3870
3871 unless (defined $_[KERNEL]->alias_resolve("Elvis")) {
3872 print "Elvis has left the building.\n";
3873 }
3874
3875As previously mentioned, alias_resolve() returns a session reference
3876or undef on failure. Failure also sets $! to ESRCH ("No such
3877process") when the ALIAS is not currently in POE::Kernel's.
3878
3879=head3 alias_list [SESSION_REFERENCE]
3880
3881alias_list() returns a list of aliases associated with a specific
3882SESSION, or with the current session if SESSION is omitted.
3883alias_list() returns an empty list if the requested SESSION has no
3884aliases.
3885
3886SESSION may be a session reference (blessed or stringified), a session
3887ID, or a session alias.
3888
3889 POE::Session->create(
3890 inline_states => {

3893 "The names I call myself: ",

3897 }
3898 );

3927=head2 I/O Watchers (Selects)
3928
3929No event system would be complete without the ability to
3930asynchronously watch for I/O events. POE::Kernel implements the
3931lowest level watchers, which are called "selects" because they were
3932historically implemented using Perl's built-in select(2) function.
3933
3934Applications handle I/O readiness events by performing some activity
3935on the underlying filehandle. Read-readiness might be handled by
3936reading from the handle. Write-readiness by writing to it.
3937
3938All I/O watcher events include two parameters. C<ARG0> contains the
3939handle that is ready for work. C<ARG1> contains an integer describing
3940what's ready.
3941
3942 sub handle_io {
3943 my ($handle, $mode) = @_[ARG0, ARG1];
3944 print "File $handle is ready for ";
3945 if ($mode == 0) {
3946 print "reading";
3947 }
3948 elsif ($mode == 1) {
3949 print "writing";
3950 }
3951 elsif ($mode == 2) {
3952 print "out-of-band reading";
3953 }
3954 else {
3955 die "unknown mode $mode";
3956 }
3957 print "\n";
3958 # ... do something here
3959 }
3960
3961The remaining parameters, C<@_[ARG2..$%_]>, contain additional
3962parameters that were passed to the POE::Kernel method that created the
3963watcher.
3964
3965POE::Kernel conditions filehandles to be 8-bit clean and non-blocking.
3966Programs that need them conditioned differently should set them up
3967after starting POE I/O watchers.
3968
3969I/O watchers will prevent sessions from stopping.

4105=head2 Session Management
4106
4107Sessions are dynamic. They may be created and destroyed during a
4108program's lifespan. When a session is created, it becomes the "child"
4109of the current session. The creator -- the current session -- becomes
4110its "parent" session. This is loosely modeled after UNIX processes.
4111
4112The most common session management is done by creating new sessions
4113and allowing them to eventually stop.
4114
4115Every session has a parent, even the very first session created.
4116Sessions without obvious parents are children of the program's
4117POE::Kernel instance.
4118
4119Child sessions will keep their parents active. See L<Session
4120Lifespans> for more about why sessions stay alive.
4121
4122The parent/child relationship tree also governs the way many signals
4123are dispatched. See L</Signal Watchers> for more information on that.
4124
4125=head3 Session Management Events (_start, _stop, _parent, _child)
4126
4127POE::Kernel provides four session management events: _start, _stop,
4128_parent and _child. They are invoked synchronously whenever a session
4129is newly created or just about to be destroyed.
4130
4131=over 2
4132
4133=item _start
4134
4135_start should be familiar by now. POE dispatches the _start event to
4136initialize a session after it has been registered under POE::Kernel.
4137What is not readily apparent, however, is that it is invoked before
4138the L<POE::Session|POE::Session> constructor returns.
4139
4140Within the _start handler, the event's sender is the session that
4141created the new session. Otherwise known as the new session's
4142I<parent>. Sessions created before POE::Kernel->run() is called will
4143be descendents of the program's POE::Kernel singleton.
4144
4145The _start handler's return value is passed to the parent session in a
4146_child event, along with the notification that the parent's new child
4147was created successfully. See the discussion of _child for more
4148details.
4149
4150 POE::Session->create(
4151 inline_states => { _start=> \&_start },
4152 args => [ $some, $args ]
4153 );
4154
4155 sub _start {
4156 my ( $some, $args ) = @_[ ARG0, ARG1 ];
4157 # ....
4158 }
4159
4160=item _stop
4161
4162_stop is a little more mysterious. POE calls a _stop handler when a
4163session is irrevocably about to be destroyed. Part of session
4164destruction is the forcible reclamation of its resources (events,
4165timers, message events, etc.) so it's not possible to post() a message
4166from _stop's handler. A program is free to try, but the event will be
4167destroyed before it has a chance to be dispatched.
4168
4169the _stop handler's return value is passed to the parent's _child
4170event. See _child for more details.
4171
4172_stop is usually invoked when a session has no further reason to live,
4173although signals may cause them to stop sooner.
4174
4175The corresponding _child handler is invoked synchronously just after
4176_stop returns.
4177
4178=item _parent
4179
4180_parent is used to notify a child session when its parent has changed.
4181This usually happens when a session is first created. It can also
4182happen when a child session is detached from its parent. See
4183L<detach_child|/"detach_child CHILD_SESSION"> and L</detach_myself>.
4184
4185_parent's ARG0 contains the session's previous parent, and ARG1
4186contains its new parent.
4187
4188 sub _parent {
4189 my ( $old_parent, $new_parent ) = @_[ ARG0, ARG1 ];
4190 print(
4191 "Session ", $_[SESSION]->ID,
4192 " parent changed from session ", $old_parent->ID,
4193 " to session ", $new_parent->ID,
4194 "\n"
4195 );
4196 }
4197
4198=item _child
4199
4200_child notifies one session when a child session has been created,
4201destroyed, or reassigned to or from another parent. It's usually
4202dispatched when sessions are created or destroyed. It can also happen
4203when a session is detached from its parent.
4204
4205_child includes some information in the "arguments" portion of @_.
4206Typically ARG0, ARG1 and ARG2, but these may be overridden by a
4207different POE::Session class:
4208

4210string may be 'create' (the child session has been created), 'gain'
4211(the child has been given by another session), or 'lose' (the child
4212session has stopped or been given away).
4213
4214In all cases, ARG1 contains a reference to the child session.
4215
4216In the 'create' case, ARG2 holds the value returned by the child
4217session's _start handler. Likewise, ARG2 holds the _stop handler's
4218return value for the 'lose' case.
4219
4220 sub _child {
4221 my( $reason, $child ) = @_[ ARG0, ARG1 ];
4222 if( $reason eq 'create' ) {
4223 my $retval = $_[ ARG2 ];
4224 }
4225 # ...
4226 }

4263

4376=head2 Signals
4377
4378POE::Kernel provides methods through which a program can register
4379interest in signals that come along, can deliver its own signals
4380without resorting to system calls, and can indicate that signals have
4381been handled so that default behaviors are not necessary.
4382
4383Signals are I<action at a distance> by nature, and their implementation
4384requires widespread synchronization between sessions (and reentrancy
4385in the dispatcher, but that's an implementation detail). Perfecting
4386the semantics has proven difficult, but POE tries to do the Right
4387Thing whenever possible.
4388
4389POE does not register %SIG handlers for signals until sig() is called
4390to watch for them. Therefore a signal's default behavior occurs for
4391unhandled signals. That is, SIGINT will gracelessly stop a program,
4392SIGWINCH will do nothing, SIGTSTP will pause a program, and so on.
4393
4394=head3 Signal Classes
4395
4396There are three signal classes. Each class defines a default behavior
4397for the signal and whether the default can be overridden. They are:
4398
4399=head4 Benign, advisory, or informative signals
4400
4401These are three names for the same signal class. Signals in this class
4402notify a session of an event but do not terminate the session if they are
4403not handled.
4404
4405It is possible for an application to create its own benign signals. See
4406L</signal> below.
4407
4408=head4 Terminal signals
4409
4410Terminal signals will kill sessions if they are not handled by a
4411L</sig_handled>() call. The OS signals that usually kill or dump a process
4412are considered terminal in POE, but they never trigger a coredump. These
4413are: HUP, INT, QUIT and TERM.
4414
4415There are two terminal signals created by and used within POE:
4416
4417=over
4418
4419=item DIE
4420
4421C<DIE> notifies sessions that a Perl exception has occurred. See
4422L</"Exception Handling"> for details.
4423
4424=item IDLE
4425
4426The C<IDLE> signal is used to notify leftover sessions that a
4427program has run out of things to do.
4428
4429=back
4430
4431=head4 Nonmaskable signals
4432
4433Nonmaskable signals are terminal regardless whether sig_handled() is
4434called. The term comes from "NMI", the non-maskable CPU interrupt
4435usually generated by an unrecoverable hardware exception.
4436
4437Sessions that receive a non-maskable signal will unavoidably stop. POE
4438implements two non-maskable signals:
4439
4440=over
4441
4442=item ZOMBIE
4443
4444This non-maskable signal is fired if a program has received an C<IDLE> signal
4445but neither restarted nor exited. The program has become a zombie (that is,
4446it's neither dead nor alive, and only exists to consume braaaains ...er...
4447memory). The C<ZOMBIE> signal acts like a cricket bat to the head,
4448bringing the zombie down, for good.
4449
4450=item UIDESTROY
4451
4452This non-maskable signal indicates that a program's user
4453interface has been closed, and the program should take the user's hint
4454and buzz off as well. It's usually generated when a particular GUI
4455widget is closed.
4456
4457=back
4458
4459=head3 Common Signal Dispatching
4460
4461Most signals are not dispatched to a single session. POE's session
4462lineage (parents and children) form a sort of family tree. When a
4463signal is sent to a session, it first passes through any children (and
4464grandchildren, and so on) that are also interested in the signal.
4465
4466In the case of terminal signals, if any of the sessions a signal passes
4467through calls L</sig_handled>(), then the signal is considered taken care
4468of. However if none of them do, then the entire session tree rooted at the
4469destination session is terminated. For example, consider this tree of
4470sessions:
4471
4472 POE::Kernel
4473 Session 2
4474 Session 4
4475 Session 5
4476 Session 3
4477 Session 6
4478 Session 7
4479
4480POE::Kernel is the parent of sessions 2 and 3. Session 2 is the
4481parent of sessions 4 and 5. And session 3 is the parent of 6 and 7.
4482
4483A signal sent to Session 2 may also be dispatched to session 4 and 5
4484because they are 2's children. Sessions 4 and 5 will only receive the
4485signal if they have registered the appropriate watcher. If the signal is
4486terminal, and none of the signal watchers in sessions 2, 4 and 5 called
4487C<sig_handled()>, all 3 sessions will be terminated.
4488
4489The program's POE::Kernel instance is considered to be a session for
4490the purpose of signal dispatch. So any signal sent to POE::Kernel
4491will propagate through every interested session in the entire program.
4492This is in fact how OS signals are handled: A global signal handler is
4493registered to forward the signal to POE::Kernel.
4494
4495=head3 Signal Semantics
4496
4497All signals come with the signal name in ARG0. The signal name is as
4498it appears in %SIG, with one exception: Child process signals are
4499always "CHLD" even if the current operating system recognizes them as
4500"CLD".
4501
4502Certain signals have special semantics:
4503
4504=head4 SIGCHLD
4505
4506=head4 SIGCLD
4507
4508Both C<SIGCHLD> and C<SIGCLD> indicate that a child process has exited
4509or been terminated by some signal. The actual signal name varies
4510between operating systems, but POE uses C<CHLD> regardless.
4511
4512Interest in C<SIGCHLD> is registered using the L</sig_child> method.
4513The L</sig>() method also works, but it's not as nice.
4514
4515The C<SIGCHLD> event includes three parameters:
4516
4517=over
4518
4519=item ARG0
4520
4521C<ARG0> contains the string 'CHLD' (even if the OS calls it SIGCLD,
4522SIGMONKEY, or something else).
4523
4524=item ARG1
4525
4526C<ARG1> contains the process ID of the finished child process.
4527
4528=item ARG2
4529
4530And C<ARG2> holds the value of C<$?> for the finished process.
4531
4532=back
4533
4534Example:
4535
4536 sub sig_CHLD {
4537 my( $name, $PID, $exit_val ) = @_[ ARG0, ARG1, ARG2 ];
4538 # ...
4539 }
4540
4541By default, SIGCHLD is not handled by registering a C<%SIG> handler.
4542Rather, waitpid() is called periodically to test for child process
4543exits. See the experimental L<USE_SIGCHLD> option if you would prefer
4544child processes to be reaped in a more timely fashion.
4545
4546=head4 SIGPIPE
4547
4548SIGPIPE is rarely used since POE provides events that do the same
4549thing. Nevertheless SIGPIPE is supported if you need it. Unlike most
4550events, however, SIGPIPE is dispatched directly to the active session
4551when it's caught. Barring race conditions, the active session should
4552be the one that caused the OS to send the signal in the first place.
4553
4554The SIGPIPE signal will still propagate to child sessions.
4555
4556ARG0 is "PIPE". There is no other information associated with this
4557signal.
4558
4559=head4 SIGWINCH
4560
4561Window resizes can generate a large number of signals very quickly.
4562This may not be a problem when using perl 5.8.0 or later, but earlier
4563versions may not take kindly to such abuse. You have been warned.
4564
4565ARG0 is "WINCH". There is no other information associated with this
4566signal.

4568=head3 Exception Handling
4569
4570POE::Kernel provides only one form of exception handling: the
4571C<DIE> signal.
4572
4573When exception handling is enabled (the default), POE::Kernel wraps state
4574invocation in C<eval{}>. If the event handler raises an exception, generally
4575with C<die>, POE::Kernel will dispatch a C<DIE> signal to the event's
4576destination session.
4577
4578C<ARG0> is the signal name, C<DIE>.
4579
4580C<ARG1> is a hashref describing the exception:
4581
4582=over
4583
4584=item error_str
4585
4586The text of the exception. In other words, C<$@>.
4587
4588=item dest_session
4589
4590Session object of the state that the raised the exception. In other words,
4591C<$_[SESSION]> in the function that died.
4592
4593=item event
4594
4595Name of the event that died.
4596
4597=item source_session
4598
4599Session object that sent the original event.
4600That is, C<$_[SENDER]> in the function that died.
4601
4602=item from_state
4603
4604State from which the original event was sent.
4605That is, C<$_[CALLER_STATE]> in the function that died.
4606
4607=item file
4608
4609Name of the file the event was sent from.
4610That is, C<$_[CALLER_FILE]> in the function that died.
4611
4612=item line
4613
4614Line number the event was sent from.
4615That is, C<$_[CALLER_LINE]> in the function that died.
4616
4617=back
4618
4619I<Note that the preceding discussion assumes you are using
4620L<POE::Session|POE::Session>'s call semantics.>
4621
4622Note that the C<DIE> signal is sent to the session that raised the
4623exception, not the session that sent the event that caused the exception to
4624be raised.
4625
4626 sub _start {
4627 $poe_kernel->sig( DIE => 'sig_DIE' );
4628 $poe_kernel->yield( 'some_event' );
4629 }
4630
4631 sub some_event {
4632 die "I didn't like that!";
4633 }
4634
4635 sub sig_DIE {
4636 my( $sig, $ex ) = @_[ ARG0, ARG1 ];
4637 # $sig is 'DIE'
4638 # $ex is the exception hash
4639 warn "$$: error in $ex->{event}: $ex->{error_str}";
4640 $poe_kernel->sig_handled();
4641
4642 # Send the signal to session that sent the original event.
4643 if( $ex->{source_session} ne $_[SESSION] ) {
4644 $poe_kernel->signal( $ex->{source_session}, 'DIE', $sig, $ex );
4645 }
4646 }

4648POE::Kernel's built-in exception handling can be disabled by setting
4649the C<POE::Kernel::CATCH_EXCEPTIONS> constant to zero. As with other

4651POE::Kernel is compiled:
4652
4653 BEGIN {
4654 package POE::Kernel;
4655 use constant CATCH_EXCEPTIONS => 0;
4656 }
4657 use POE;
4658
4659or
4660

4668=head3 sig SIGNAL_NAME [, EVENT_NAME]
4669
4670sig() registers or unregisters an EVENT_NAME event for a particular
4671SIGNAL_NAME. The event is registered if EVENT_NAME is defined, otherwise
4672the SIGNAL_NAME handler is unregistered. This means that a session can
4673register only one handler per SIGNAL_NAME; subsequent registration attempts
4674will replace the old handler.
4675
4676SIGNAL_NAMEs are generally the same as members of C<%SIG>, with two
4677exceptions. First, C<CLD> is an alias for C<CHLD> (although see
4678L</sig_child>). And second, it's possible to send and handle signals
4679created by the application and have no basis in the operating system.
4680
4681 sub handle_start {
4682 $_[KERNEL]->sig( INT => "event_ui_shutdown" );
4683 $_[KERNEL]->sig( bat => "holy_searchlight_batman" );
4684 $_[KERNEL]->sig( signal => "main_screen_turn_on" );
4685 }
4686
4687The operating system may never be able to generate the last two
4688signals, but a POE session can by using POE::Kernel's
4689L</signal>() method.
4690
4691Later on the session may decide not to handle the signals:
4692
4693 sub handle_ui_shutdown {
4694 $_[KERNEL]->sig( "INT" );
4695 $_[KERNEL]->sig( "bat" );
4696 $_[KERNEL]->sig( "signal" );
4697 }
4698
4699More than one session may register interest in the same signal, and a
4700session may clear its own signal watchers without affecting those in
4701other sessions.
4702
4703sig() does not return a meaningful value.
4704
4705=head3 sig_child PROCESS_ID [, EVENT_NAME]
4706
4707sig_child() is a convenient way to deliver an EVENT_NAME event when a
4708particular PROCESS_ID has exited. The watcher can be cleared
4709prematurely by calling sig_child() with just the PROCESS_ID.
4710
4711A session may register as many sig_child() handlers as necessary, but
4712a session may only have one per PROCESS_ID.
4713
4714sig_child() watchers are one-shot. They automatically unregister
4715themselves once the EVENT_NAME has been delivered.
4716
4717sig_child() watchers keep a session alive for as long as they are
4718active. This is unique among signal watchers.
4719
4720Programs that wish to reliably reap child processes should be sure to
4721call sig_child() before returning from the event handler that forked
4722the process. Otherwise POE::Kernel may have an opportunity to call
4723waitpid() before an appropriate event watcher has been registered.
4724
4725sig_child() does not return a meaningful value.
4726
4727 sub forked_parent {
4728 my( $heap, $pid, $details ) = @_[ HEAP, ARG0, ARG1 ];
4729 $heap->{$pid} = $details;
4730 $poe_kernel->sig_child( $pid, 'sig_child' );
4731 }
4732
4733 sub sig_child {
4734 my( $heap, $sig, $pid, $exit_val ) = @_[ HEAP, ARG0, ARG1, ARG2 ];
4735 my $details = delete $heap->{ $pid };
4736 warn "$$: Child $pid exited"
4737 # ....
4738 }

4740=head3 sig_handled
4741
4742sig_handled() informs POE::Kernel that the currently dispatched signal has
4743been handled by the currently active session. If the signal is terminal, the
4744sig_handled() call prevents POE::Kernel from stopping the sessions that
4745received the signal.
4746
4747A single signal may be dispatched to several sessions. Only one needs
4748to call sig_handled() to prevent the entire group from being stopped.
4749If none of them call it, however, then they are all stopped together.
4750
4751sig_handled() does not return a meaningful value.
4752
4753 sub _start {
4754 $_[KERNEL]->sig( INT => 'sig_INT' );
4755 }
4756

4762=head3 signal SESSION, SIGNAL_NAME [, ARGS_LIST]
4763
4764signal() posts a SIGNAL_NAME signal to a specific SESSION with an
4765optional ARGS_LIST that will be passed to every interested handler. As
4766mentioned elsewhere, the signal may be delivered to SESSION's
4767children, grandchildren, and so on. And if SESSION is the POE::Kernel
4768itself, then all interested sessions will receive the signal.
4769
4770It is possible to send a signal in POE that doesn't exist in the
4771operating system. signal() places the signal directly into POE's
4772event queue as if they came from the operating system, but they are
4773not limited to signals recognized by kill(). POE uses a few of these
4774fictitious signals for its own global notifications.
4775

4777
4778 sub some_event_handler {
4779 # Turn on all main screens.
4780 $_[KERNEL]->signal( $_[KERNEL], "signal" );

4784setting $! to explain the nature of the failure:
4785
4786=over
4787
4788=item ESRCH ("No such process")
4789
4790The SESSION does not exist.
4791
4792=back

4795the kernel will propagate the signal to all sessions. This is a cheap

4888=head2 Public Reference Counters
4889
4890The methods in this section manipulate reference counters on the
4891current session or another session.
4892
4893Each session has a namespace for user-manipulated reference counters.
4894These namespaces are associated with the target SESSION_ID for the
4895reference counter methods, not the caller. Nothing currently prevents
4896one session from decrementing a reference counter that was incremented
4897by another, but this behavior is not guaranteed to remain. For now,
4898it's up to the users of these methods to choose obscure counter names
4899to avoid conflicts.
4900
4901Reference counting is a big part of POE's magic. Various objects
4902(mainly event watchers and components) hold references to the sessions
4903that own them. L</Session Lifespans> explains the concept in more
4904detail.
4905
4906The ability to keep a session alive is sometimes useful in an application or
4907library. For example, a component may hold a public reference to another
4908session while it processes a request from that session. In doing so, the
4909component guarantees that the requester is still around when a response is
4910eventually ready. Keeping a reference to the session's object is not
4911enough. POE::Kernel has its own internal reference counting mechanism.
4912
4913=head3 refcount_increment SESSION_ID, COUNTER_NAME
4914
4915refcount_increment() increases the value of the COUNTER_NAME reference
4916counter for the session identified by a SESSION_ID. To discourage the
4917use of session references, the refcount_increment() target session
4918must be specified by its session ID.
4919
4920The target session will not stop until the value of any and all of its
4921COUNTER_NAME reference counters are zero. (Actually, it may stop in
4922some cases, such as failing to handle a terminal signal.)
4923
4924Negative reference counters are legal. They still must be incremented
4925back to zero before a session is eligible for stopping.
4926
4927 sub handle_request {
4928 # Among other things, hold a reference count on the sender.
4929 $_[KERNEL]->refcount_increment( $_[SENDER]->ID, "pending request");
4930 $_[HEAP]{requesters}{$request_id} = $_[SENDER]->ID;
4931 }
4932
4933For this to work, the session needs a way to remember the
4934$_[SENDER]->ID for a given request. Customarily the session generates
4935a request ID and uses that to track the request until it is fulfilled.
4936
4937refcount_increment() returns true on success or false on failure.
4938Furthermore, $! is set on failure to one of:
4939
4940ESRCH: The SESSION_ID does not refer to a currently active session.
4941
4942=head3 refcount_decrement SESSION_ID, COUNTER_NAME
4943
4944refcount_decrement() reduces the value of the COUNTER_NAME reference
4945counter for the session identified by a SESSION_ID. It is the
4946counterpoint for refcount_increment(). Please see
4947refcount_increment() for more context.
4948
4949 sub finally_send_response {
4950 # Among other things, release the reference count for the
4951 # requester.
4952 my $requester_id = delete $_[HEAP]{requesters}{$request_id};
4953 $_[KERNEL]->refcount_decrement( $requester_id, "pending request");
4954 }
4955
4956The requester's $_[SENDER]->ID is remembered and removed from the heap
4957(lest there be memory leaks). It's used to decrement the reference
4958counter that was incremented at the start of the request.
4959
4960refcount_decrement() returns true on success or false on failure.
4961Furthermore, $! is set on failure to one of:
4962
4963ESRCH: The SESSION_ID does not refer to a currently active session.
4964
4965It is not possible to discover currently active public references. See
4966L<POE::API::Peek>.

5083=head1 DEBUGGING POE AND PROGRAMS USING IT
5084
5085POE includes quite a lot of debugging code, in the form of both fatal
5086assertions and run-time traces. They may be enabled at compile time,
5087but there is no way to toggle them at run-time. This was done to
5088avoid run-time penalties in programs where debugging is not necessary.
5089That is, in most production cases.
5090
5091Traces are verbose reminders of what's going on within POE. Each is
5092prefixed with a four-character field describing the POE subsystem that
5093generated it.
5094
5095Assertions (asserts) are quiet but deadly, both in performance (they
5096cause a significant run-time performance hit) and because they cause
5097fatal errors when triggered.
5098
5099The assertions and traces are useful for developing programs with POE,
5100but they were originally added to debug POE itself.
5101
5102Each assertion and tracing group is enabled by setting a constant in
5103the POE::Kernel namespace to a true value. This is the same mechanism
5104documented under L</"Using Time::HiRes">, namely:
5105
5106 BEGIN {
5107 package POE::Kernel;
5108 use constant ASSERT_DEFAULT => 1;
5109 }
5110 use POE;
5111
5112or
5113
5114 sub POE::Kernel::ASSERT_DEFAULT () { 1 }
5115 use POE;
5116
5117As mentioned in L</"Using Time::HiRes">, the switches must be defined as
5118constants before POE::Kernel is first loaded. Otherwise Perl's
5119compiler will not see the constants when first compiling POE::Kernel,
5120and the features will not be properly enabled.
5121
5122Assertions and traces may also be enabled by setting shell environment
5123variables. The environment variables are named after the POE::Kernel
5124constants with a "POE_" prefix.
5125
5126 POE_ASSERT_DEFAULT=1 POE_TRACE_DEFAULT=1 ./my_poe_program
5127
5128In alphabetical order:
5129
5130=head2 ASSERT_DATA
5131
5132ASSERT_DATA enables run-time data integrity checks within POE::Kernel
5133and the classes that mix into it. POE::Kernel tracks a lot of
5134cross-referenced data, and this group of assertions ensures that it's
5135consistent.
5136
5137Prefix: <dt>
5138
5139Environment variable: POE_ASSERT_DATA
5140
5141=head2 ASSERT_DEFAULT
5142
5143ASSERT_DEFAULT specifies the default value for assertions that are not
5144explicitly enabled or disabled. This is a quick and reliable way to
5145make sure all assertions are on.
5146
5147No assertion uses ASSERT_DEFAULT directly, and this assertion flag has
5148no corresponding output prefix.
5149
5150Turn on all assertions except ASSERT_EVENTS:
5151
5152 sub POE::Kernel::ASSERT_DEFAULT () { 1 }
5153 sub POE::Kernel::ASSERT_EVENTS () { 0 }
5154 use POE::Kernel;
5155
5156Prefix: (none)
5157
5158Environment variable: POE_ASSERT_DEFAULT
5159
5160=head2 ASSERT_EVENTS
5161
5162ASSERT_EVENTS mainly checks for attempts to dispatch events to
5163sessions that don't exist. This assertion can assist in the debugging
5164of strange, silent cases where event handlers are not called.
5165
5166Prefix: <ev>
5167
5168Environment variable: POE_ASSERT_EVENTS
5169
5170=head2 ASSERT_FILES
5171

5175Prefix: <fh>

5394=head2 USE_SIGNAL_PIPE
5395
5396The only safe way to handle signals is to implement a shared-nothing
5397model. POE builds a I<signal pipe> that communicates between the

5399The signal pipe is implemented with L<POE::Pipe::OneWay>, using a
5400C<pipe> conduit on Unix. Unfortunately, the signal pipe is not compatible
5401with Windows and is not used on that platform.

5404must set C<USE_SIGNAL_PIPE> to 0, or the environment variable
5405C<POE_USE_SIGNAL_PIPE>.
