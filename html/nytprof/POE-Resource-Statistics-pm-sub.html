208=head1 NAME
209
210POE::Resource::Statistics -- experimental run-time statistics for POE
211
212=head1 SYNOPSIS
213
214 my %stats = $poe_kernel->stat_getdata;
215 printf "Idle = %3.2f\n", 100*$stats{avg_idle_seconds}/$stats{interval};
216
217=head1 DESCRIPTION
218
219POE::Resource::Statistics is a mix-in class for POE::Kernel. It
220provides features for gathering run-time statistics about POE::Kernel
221and the applications that use it.
222
223Statistics gathering is enabled with the TRACE_STATISTICS constant.
224There is no run-time performance penalty when tracing is disabled.
225
226Statistics are totaled every 30 seconds, and a rolling average is
227maintained for the last two minutes. The data may be retrieved at any
228time with the stat_getdata() method. Statistics will also be
229displayed on the console shortly before POE::Kernel's run() returns.
230
231The time() function is used to gather statistics over time. If
232Time::HiRes is available, it will be used automatically. Otherwise
233time is measured in whole seconds, and the resulting rounding errors
234will make the statistics much less useful.
235
236Run-time statistics gathering was added to POE 0.28. It is considered
237B<highly experimental>. Please be advised that the statistics it
238gathers are quite likely wrong. They may in fact be useless. The
239reader is invited to investigate and improve the module's
240methodologies.
241
242=head1 Gathered Statistics
243
244stat_getdata() returns a hashref with a small number of accumulated
245values. For each accumulator, there will be a corresponding field
246prefixed "avg_" which is the rolling average for that accumulator.
247
248=head2 blocked
249
250C<blocked> contains the number of events (both user and kernel) which
251were delayed due to a user event running for too long. On conclusion
252of the program, POE will display the blocked count.
253
254In theory, one can compare C<blocked> with C<user_events> to determine
255how much lag is produced by user code. C<blocked> should be as low as
256possible to ensure minimum user-generated event lag.
257
258In practice, C<blocked> is often near or above C<user_events>. Events
259that are even the slightest bit late count as being "blocked". See
260C<blocked_seconds> for a potentially more useful metric.
261
262=head2 blocked_seconds
263
264C<blocked_seconds> contains the total number of seconds that events
265waited in the queue beyond their nominal dispatch times. The average
266(C<avg_blocked_seconds>) is generally more useful.
267
268=head2 idle_seconds
269
270C<idle_seconds> is the amount of time that POE spent doing nothing at
271all. Typically this time is spent waiting for I/O or timers rather
272than dispatching events.
273
274=head2 interval
275
276C<interval> is the average interval over which the statistics counters
277are recorded. This will typically be 30 seconds, but it can be more
278if long-running user events prevent statistics from being gathered on
279time. C<interval> may also be less for programs that finish in under
280half a minute.
281
282C<avg_interval> may often be lower, as the very last measurement taken
283before POE::Kernel's run() returns will almost always have an
284C<interval> less than 30 seconds.
285
286=head2 total_duration
287
288C<total_duration> contains the total time over which the average was
289calculated. The "avg_" accumulators are averaged over a 2-minute
290interval. C<total_duration> may vary from 120 seconds due to the same
291reasons as described in L</interval>.
292
293=head2 user_events
294
295C<user_events> contains the number of events that have been dispatched
296to user code. "User" events do not include POE's internal events,
297although it will include events dispatched on behalf of wheels.
298
299Shortly before POE::Kernel's run() returns, a C<user_load> value will
300be computed showing the average number of user events that have been
301dispatched per second. A very active web server should have a high
302C<user_load> value. The higher the user load, the more important it
303is to have small C<blocked> and C<blocked_seconds> values.
304
305=head2 user_seconds
306
307C<user_seconds> is the time that was spent handling user events,
308including those handled by wheels. C<user_seconds> + C<idle_seconds>
309should typically add up to C<total_duration>. Any difference is
310unaccounted time in POE, and indicates a flaw in the statistics
311gathering methodology.
312
313=head1 SEE ALSO
314
315See L<POE::Kernel/TRACE_STATISTICS> for instructions to enable
316statistics gathering.
317
318=head1 BUGS
319
320Statistics may be highly inaccurate. This feature is B<highly
321experimental> and may change significantly over time.
