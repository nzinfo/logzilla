310=head1 NAME
311
312Log::Log4perl::Appender - Log appender class
313
314=head1 SYNOPSIS
315
316 use Log::Log4perl;
317
318 # Define a logger
319 my $logger = Log::Log4perl->get_logger("abc.def.ghi");
320
321 # Define a layout
322 my $layout = Log::Log4perl::Layout::PatternLayout->new(
323 "%d (%F:%L)> %m");
324
325 # Define an appender
326 my $appender = Log::Log4perl::Appender->new(
327 "Log::Log4perl::Appender::Screen",
328 name => 'dumpy');
329
330 # Set the appender's layout
331 $appender->layout($layout);
332 $logger->add_appender($appender);
333
334=head1 DESCRIPTION
335
336This class is a wrapper around the C<Log::Log4perl::Appender>
337appender set.
338
339It also supports the <Log::Dispatch::*> collections of appenders. The
340module hides the idiosyncrasies of C<Log::Dispatch> (e.g. every
341dispatcher gotta have a name, but there's no accessor to retrieve it)
342from C<Log::Log4perl> and yet re-uses the extremely useful variety of
343dispatchers already created and tested in C<Log::Dispatch>.
344
345=head1 FUNCTIONS
346
347=head2 Log::Log4perl::Appender->new($dispatcher_class_name, ...);
348
349The constructor C<new()> takes the name of the appender
350class to be created as a I<string> (!) argument, optionally followed by
351a number of appender-specific parameters,
352for example:
353
354 # Define an appender
355 my $appender = Log::Log4perl::Appender->new(
356 "Log::Log4perl::Appender::File"
357 filename => 'out.log');
358
359In case of C<Log::Dispatch> appenders,
360if no C<name> parameter is specified, the appender object will create
361a unique one (format C<appNNN>), which can be retrieved later via
362the C<name()> method:
363
364 print "The appender's name is ", $appender->name(), "\n";
365
366Other parameters are specific to the appender class being used.
367In the case above, the C<filename> parameter specifies the name of
368the C<Log::Log4perl::Appender::File> dispatcher used.
369
370However, if, for instance,
371you're using a C<Log::Dispatch::Email> dispatcher to send you
372email, you'll have to specify C<from> and C<to> email addresses.
373Every dispatcher is different.
374Please check the C<Log::Dispatch::*> documentation for the appender used
375for details on specific requirements.
376
377The C<new()> method will just pass these parameters on to a newly created
378C<Log::Dispatch::*> object of the specified type.
379
380When it comes to logging, the C<Log::Log4perl::Appender> will transparently
381relay all messages to the C<Log::Dispatch::*> object it carries
382in its womb.
383
384=head2 $appender->layout($layout);
385
386The C<layout()> method sets the log layout
387used by the appender to the format specified by the
388C<Log::Log4perl::Layout::*> object which is passed to it as a reference.
389Currently there's two layouts available:
390
391 Log::Log4perl::Layout::SimpleLayout
392 Log::Log4perl::Layout::PatternLayout
393
394Please check the L<Log::Log4perl::Layout::SimpleLayout> and
395L<Log::Log4perl::Layout::PatternLayout> manual pages for details.

442=head1 Pitfalls
443
444Since the C<Log::Dispatch::File> appender truncates log files by default,
445and most of the time this is I<not> what you want, we've instructed

449
450 log4perl.category = INFO, FileAppndr

460 log4perl.appender.FileAppndr.filename = test.log
461 log4perl.appender.FileAppndr.mode = write
462 log4perl.appender.FileAppndr.layout = Log::Log4perl::Layout::SimpleLayout
463
464it will overwrite an existing log file C<test.log> and start from scratch.

466=head1 Appenders Expecting Message Chunks
467
468Instead of simple strings, certain appenders are expecting multiple fields
469as log messages. If a statement like
470
471 $logger->debug($ip, $user, "signed in");
472
473causes an off-the-shelf C<Log::Log4perl::Appender::Screen>
474appender to fire, the appender will
475just concatenate the three message chunks passed to it
476in order to form a single string.
477The chunks will be separated by a string defined in
478C<$Log::Log4perl::JOIN_MSG_ARRAY_CHAR> (defaults to the empty string
479"").
480
481However, different appenders might choose to
482interpret the message above differently: An
483appender like C<Log::Log4perl::Appender::DBI> might take the
484three arguments passed to the logger and put them in three separate
485rows into the DB.
486
487The C<warp_message> appender option is used to specify the desired
488behavior.
489If no setting for the appender property
490
491 # *** Not defined ***
492 # log4perl.appender.SomeApp.warp_message
493
494is defined in the Log4perl configuration file, the
495appender referenced by C<SomeApp> will fall back to the standard behavior
496and join all message chunks together, separating them by
497C<$Log::Log4perl::JOIN_MSG_ARRAY_CHAR>.
498
499If, on the other hand, it is set to a false value, like in
500
501 log4perl.appender.SomeApp.layout=NoopLayout
502 log4perl.appender.SomeApp.warp_message = 0
503
504then the message chunks are passed unmodified to the appender as an
505array reference. Please note that you need to set the appender's
506layout to C<Log::Log4perl::Layout::NoopLayout> which just leaves
507the messages chunks alone instead of formatting them or replacing
508conversion specifiers.
509
510B<Please note that the standard appenders in the Log::Dispatch hierarchy
511will choke on a bunch of messages passed to them as an array reference.
512You can't use C<warp_message = 0> (or the function name syntax
513defined below) on them.
514Only special appenders like Log::Log4perl::Appender::DBI can deal with
515this.>
516
517If (and now we're getting fancy)
518an appender expects message chunks, but we would
519like to pre-inspect and probably modify them before they're
520actually passed to the appender's C<log>
521method, an inspection subroutine can be defined with the
522appender's C<warp_message> property:
523
524 log4perl.appender.SomeApp.layout=NoopLayout
525 log4perl.appender.SomeApp.warp_message = sub { \
526 $#_ = 2 if @_ > 3; \
527 return @_; }
528
529The inspection subroutine defined by the C<warp_message>
530property will receive the list of message chunks, like they were
531passed to the logger and is expected to return a corrected list.
532The example above simply limits the argument list to a maximum of
533three by cutting off excess elements and returning the shortened list.
534
535Also, the warp function can be specified by name like in
536
537 log4perl.appender.SomeApp.layout=NoopLayout
538 log4perl.appender.SomeApp.warp_message = main::filter_my_message
539
540In this example,
541C<filter_my_message> is a function in the C<main> package,
542defined like this:
543
544 my $COUNTER = 0;
545
546 sub filter_my_message {
547 my @chunks = @_;
548 unshift @chunks, ++$COUNTER;
549 return @chunks;
550 }
551
552The subroutine above will add an ever increasing counter
553as an additional first field to
554every message passed to the C<SomeApp> appender -- but not to
555any other appender in the system.

557=head2 Composite Appenders
558
559Composite appenders relay their messages to sub-appenders after providing
560some filtering or synchronizing functionality on incoming messages.
561Examples are
562Log::Log4perl::Appender::Synchronized,
563Log::Log4perl::Appender::Limit, and
564Log::Log4perl::Appender::Buffer. Check their manual pages for details.
565
566Composite appender objects are regular Log::Log4perl::Appender objects,
567but they have the composite flag set:
568
569 $app->composite(1);
570
571and they define a post_init() method, which sets the appender it relays
572its messages to:
573
574 ###########################################
575 sub post_init {
576 ############################################
577 my($self) = @_;
578
579 if(! exists $self->{appender}) {
580 die "No appender defined for " . __PACKAGE__;
581 }
582
583 my $appenders = Log::Log4perl->appenders();
584 my $appender = Log::Log4perl->appenders()->{$self->{appender}};
585
586 if(! defined $appender) {
587 die "Appender $self->{appender} not defined (yet) when " .
588 __PACKAGE__ . " needed it";
589 }
590
591 $self->{app} = $appender;
592 }
593
594The reason for this post-processing step is that the relay appender
595might not be defined yet when the composite appender gets defined.
596This can happen if Log4perl is initialized with a configuration file
597(which is the most common way to initialize Log4perl), because
598appenders spring into existance in unpredictable order.
599
600For example, if you define a Synchronized appender like
601
602 log4perl.appender.Syncer = Log::Log4perl::Appender::Synchronized

606I<name> of the appender to finally relay messages to. After the
607Log4perl configuration file has been processed, Log4perl will remember to
608call the composite appender's post_init() method, which will grab
609the relay appender instance referred to by the name (Logfile)
610and set it in its C<app> attribute. This is exactly what the
611code snippet above does.

613But if you initialize Log4perl by its API, you need to remember to
614perform these steps. Here's the lineup:
615
616 use Log::Log4perl qw(get_logger :levels);
617
618 my $fileApp = Log::Log4perl::Appender->new(
619 'Log::Log4perl::Appender::File',
620 name => 'MyFileApp',
621 filename => 'mylog',
622 mode => 'append',
623 );
624 $fileApp->layout(
625 Log::Log4perl::Layout::PatternLayout::Multiline->new(
626 '%d{yyyy-MM-dd HH:mm:ss} %p [%c] #%P> %m%n')
627 );
628 # Make the appender known to the system (without assigning it to
629 # any logger
630 Log::Log4perl->add_appender( $fileApp );
631
632 my $syncApp = Log::Log4perl::Appender->new(
633 'Log::Log4perl::Appender::Synchronized',
634 name => 'MySyncApp',
635 appender => 'MyFileApp',
636 key => 'nem',
637 );
638 $syncApp->post_init();
639 $syncApp->composite(1);
640
641 # The Synchronized appender is now ready, assign it to a logger
642 # and start logging.
643 get_logger("")->add_appender($syncApp);
644
645 get_logger("")->level($DEBUG);
646 get_logger("wonk")->debug("waah!");

648The composite appender's log() function will typically cache incoming
649messages until a certain trigger condition is met and then forward a bulk
650of messages to the relay appender.
651
652Caching messages is surprisingly tricky, because you want them to look
653like they came from the code location they were originally issued from
654and not from the location that triggers the flush. Luckily, Log4perl
655offers a cache mechanism for messages, all you need to do is call the
656base class' log() function with an additional reference to a scalar,
657and then save its content to your composite appender's message buffer
658afterwards:
659
660 ###########################################
661 sub log {
662 ###########################################
663 my($self, %params) = @_;
664
665 # ... some logic to decide whether to cache or flush
666
667 # Adjust the caller stack
668 local $Log::Log4perl::caller_depth =
669 $Log::Log4perl::caller_depth + 2;
670
671 # We need to cache.
672 # Ask the appender to save a cached message in $cache
673 $self->{relay_app}->SUPER::log(\%params,
674 $params{log4p_category},
675 $params{log4p_level}, \my $cache);
676
677 # Save it in the appender's message buffer
678 push @{ $self->{buffer} }, $cache;
679 }
680
681Note that before calling the log() method of the relay appender's base class
682(and thus introducing two additional levels on the call stack), we need to
683adjust the call stack to allow Log4perl to render cspecs like the %M or %L
684correctly. The cache will then contain a correctly rendered message, according
685to the layout of the target appender.
686
687Later, when the time comes to flush the cached messages, a call to the relay
688appender's base class' log_cached() method with the cached message as
689an argument will forward the correctly rendered message:
690
691 ###########################################
692 sub log {
693 ###########################################
694 my($self, %params) = @_;
695
696 # ... some logic to decide whether to cache or flush
697
698 # Flush pending messages if we have any
699 for my $cache (@{$self->{buffer}}) {
700 $self->{relay_app}->SUPER::log_cached($cache);
701 }
702 }
